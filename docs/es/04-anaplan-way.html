La metodología de delivery de IFP en seis fases

El delivery de IFP sigue las fases estándar de delivery de aplicaciones de Anaplan, adaptadas al flujo de trabajo de generación del Application Framework Configurator. Entender las fases le ayuda a fijar expectativas correctas con los clientes y saber qué actividades pertenecen a cada lugar.

📚 Documento de referencia

The Anaplan Way for Applications Methodology: metodología completa para entregar implementaciones de aplicaciones estructuradas.

1

Pre-Project

La fase de preparación del compromiso. Los entregables incluyen:

  • Session Zero: descubrimiento de configuración con el cliente
  • Hoja de trabajo de configuración completada (las 44 preguntas respondidas)
  • Nombrado de Jerarquías y número de niveles determinados
  • Disponibilidad de datos confirmada: fuente de actuals, feed de headcount, configuración de moneda
  • Workspace aprovisionado, acceso de Workspace Admin confirmado para el equipo de delivery

2

Config

La fase del Application Framework Configurator. El consultor de delivery:

  • Ingresa todas las respuestas de configuración en el Application Framework Configurator
  • Revisa preguntas de nivel superior; financial planning; headcount; configuraciones de capex
  • Dispara la generación del modelo
  • Revisa los Error Logs

3

Test & Deploy

Fase de validación después de una generación limpia:

  • Poblar Listas/Jerarquías y cargar datos de actuals de muestra vía pipelines ADO
  • Verificar puntualmente los Line Items calculados en los Módulos CALC
  • Validar los métodos de planeación en las páginas UX
  • UAT con usuarios clave del equipo del cliente

4

Extension Foundation

Diseñar y dimensionar las extensiones específicas del cliente:

  • Identificar brechas entre el modelo base IFP y los requisitos del cliente
  • Clasificar cada brecha: Configurar vs. Extender vs. Modificar
  • Dimensionar cada extensión (S/M/L/XL)
  • Crear Data Tags de extensión en el modelo (Extension + Formula Override)

5

Iterative Extension

Construir extensiones en orden de prioridad:

  • Cada extensión: diseñar → construir → probar → documentar
  • Marcar cada Line Item que añada o modifique
  • Documentar en una especificación de diseño externa (no solo notas en el modelo)
  • Pruebas de regresión de la funcionalidad del modelo base después de cada extensión

6

Extension Test & Deploy

Validación final y entrega:

  • UAT completo con todas las extensiones en su lugar
  • Prueba de rendimiento con volúmenes de datos de producción
  • Documentación de entrega: log de extensiones, mapa de formula override, lista de verificación de actualización
  • Capacitación para los admins del cliente

ℹ Notas de fase específicas de IFP

Session Zero es crítica. A diferencia de las construcciones de modelos personalizados, las decisiones de configuración de IFP tomadas en Session Zero son difíciles o imposibles de cambiar después de la generación. Los niveles de Jerarquía, la estructura geográfica y la configuración de entidad deben fijarse antes de que alguien toque el Application Framework Configurator. Una Dimensión faltante en Session Zero = una regeneración completa durante Test & Deploy.

Dos tipos de cliente

🌱 Greenfield

  • Workspace nuevo, sin modelos Anaplan existentes
  • IFP es la primera aplicación
  • Sin complejidad de integración con modelos existentes
  • Configuración completa de ADO desde cero
  • Riesgo: el descubrimiento de Session Zero es la única restricción; hágalo bien

🔗 Workspace existente

  • IFP desplegado junto a modelos en vivo existentes
  • El Workspace Admin debe gestionar conflictos de Listas
  • ADO puede necesitar coexistir con import actions existentes
  • La nomenclatura de Listas no debe colisionar con Listas existentes
  • Riesgo: la generación puede afectar los ajustes del Workspace existente
