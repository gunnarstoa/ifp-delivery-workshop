Esta é uma visão geral do Anaplan Way for Applications e não substitui a sessão de treinamento completa sobre a metodologia abrangente. Use esta página para se orientar sobre as seis fases e onde o trabalho de entrega do IFP se encaixa — não como substituto do curso completo de metodologia.

A metodologia de entrega do IFP em seis fases

A entrega do IFP segue as fases padrão de entrega de App da Anaplan, adaptadas ao fluxo de geração orientada pelo Application Framework Configurator. Entender as fases ajuda você a definir as expectativas corretas com os clientes e saber quais atividades pertencem a onde.

📚 Documento de referência

The Anaplan Way for Applications Methodology — Metodologia completa para entrega de implementações de aplicações estruturadas.

1

Pré-projeto

A fase de preparação do projeto. Entregáveis incluem:

  • Session Zero — descoberta de configuração com o cliente
  • Worksheet de configuração concluído (todas as 44 perguntas respondidas)
  • Nomenclatura das Hierarquias e número de níveis determinados
  • Disponibilidade de dados confirmada: fonte de actuals, feed de headcount, configuração de moeda
  • Workspace provisionado, acesso de Workspace Admin confirmado para a equipe de entrega

2

Config

A fase do Application Framework Configurator. O consultor de entrega:

  • Insere todas as respostas de configuração no Application Framework Configurator
  • Revisa perguntas de nível superior; configurações de financial planning; headcount; capex
  • Dispara a Geração do modelo
  • Revisa Error Logs

3

Test & Deploy

Fase de validação após uma Geração limpa:

  • Popular Listas/Hierarquias e carregar dados de actuals de amostra via pipelines ADO
  • Verificar por amostragem Line Items calculados nos Módulos CALC
  • Validar métodos de planejamento nas páginas UX
  • UAT com usuários-chave da equipe do cliente

4

Extension Foundation

Projetar e escopar extensões específicas do cliente:

  • Identificar gaps entre o modelo IFP base e os requisitos do cliente
  • Classificar cada gap: Configurar vs. Estender vs. Modificar
  • Dimensionar cada extensão (S/M/L/XL)
  • Criar Data Tags de extensão no modelo (Extension + Formula Override)

5

Iterative Extension

Construir extensões em ordem de prioridade:

  • Cada extensão: design → construção → teste → documentação
  • Marcar cada Line Item que você adicionar ou modificar
  • Documentar em spec de design externo (não apenas notas dentro do modelo)
  • Regression test da funcionalidade do modelo base após cada extensão

6

Extension Test & Deploy

Validação final e handover:

  • UAT completo com todas as extensões implementadas
  • Teste de desempenho com volumes de dados de produção
  • Documentação de handover: log de extensões, mapa de formula override, checklist de upgrade
  • Treinamento para admins do cliente

ℹ Notas específicas de fases do IFP

A Session Zero é crítica. Diferentemente de construções de modelo personalizadas, decisões de configuração do IFP feitas na Session Zero são difíceis ou impossíveis de mudar após a Geração. Níveis de Hierarquia, estrutura de geografia e configuração de entity precisam ser travados antes que alguém toque no Application Framework Configurator. Uma Dimensão esquecida na Session Zero = uma regeneração completa durante Test & Deploy.

Dois tipos de cliente

🌱 Greenfield

  • Novo Workspace, sem modelos Anaplan existentes
  • IFP é a primeira aplicação
  • Sem complexidade de integração com modelos existentes
  • Configuração completa do ADO do zero
  • Risco: a descoberta da Session Zero é a única restrição — acerte de primeira

🔗 Workspace existente

  • IFP implantado ao lado de modelos live existentes
  • O Workspace Admin precisa gerenciar conflitos de Listas
  • O ADO pode precisar coexistir com ações de import existentes
  • A nomenclatura de Listas não pode colidir com Listas existentes
  • Risco: a Geração pode afetar configurações existentes do Workspace
