Error Logs — O que são

Error Logs são gerados pelo Application Framework após a Geração do modelo. Eles listam questões de fórmula e configuração que surgiram quando o modelo gerado difere do template do modelo base. O Error Log é acessível pelo canto inferior direito da página de visão geral do Application Framework e pode ser exportado para Excel.

Error Logs são conteúdo novo introduzido no IFP 2.0 — versões anteriores do IFP não tinham esse relatório sistemático de erros. Aprender a lê-los com eficiência é uma habilidade-chave de praticante.

Referência de tipos de erro

Tipo de erroO que significaAção
LINE_ITEM_FORMULAUma fórmula referencia um Line Item que não existe ou foi renomeado no modelo gerado — frequentemente devido a renomeação de Hierarquia ou escolhas de configuraçãoEncontre o Line Item pelo código (prefixo underscore), compare a fórmula com o modelo base, corrija ou copie-ajuste do base

IMPORT_ACTIONUma ação de import referencia um Módulo de origem ou Lista que mudouAtualize o mapeamento de import no modelo afetado
MODEL_DATA_SOURCEUma conexão de fonte de dados está quebrada (referência de pipeline ADO). Fontes de dados de modelo externas também não são reconhecidas.Restabeleça a conexão no ADO ou atualize a referência da fonte de dados

Como encontrar um erro de fórmula

O tipo de erro mais comum é LINE_ITEM_FORMULA. Aqui está o fluxo diagnóstico:

"Geralmente indicamos o Line Item com um underscore no começo — esse é o código do Line Item com erro. Entre no modelo, procure por esse código e ele leva você direto ao item exato."
— Orientação do workshop
1

Abra o Error Log

Na visão geral do Application Framework, clique no painel de Error Log (canto inferior direito). Filtre apenas para erros de fórmula. Se mais de 10 erros, exporte para Excel primeiro.

2

Anote o código

Cada erro inclui um código de Line Item no formato _LINEITEMNAMEEXCEPTION (prefixo underscore). Esse código é único — use-o para a busca no modelo, não o nome de exibição.

3

Encontre o Line Item afetado

Vá para o modelo afetado (o Error Log diz qual modelo). Abra Line Items. Procure pelo código com o prefixo underscore. O resultado da busca leva você diretamente ao Módulo e ao Line Item.

4

Compare com o modelo base

Abra o modelo base TRAIN (já provisionado no seu tenant). Encontre o mesmo Line Item. Compare a fórmula gerada com a fórmula do modelo base. Identifique o que mudou: renomeação de Hierarquia, remoção de geografia, mudança de Dimensão.

5

Corrija ou copie-ajuste

Opção A: Corrija a fórmula diretamente — atualize as referências para combinar com sua configuração. Opção B: Copie a fórmula do modelo base para o seu modelo e ajuste as referências de Dimensão. A Opção B é mais segura e rápida para a maioria dos erros LINE_ITEM_FORMULA.

6

Valide

Após corrigir, verifique se a fórmula resolve (sem indicador vermelho de erro de fórmula). Volte ao Error Log e marque o erro como resolvido. Prossiga para o próximo erro.

7

Escale se travar

Se o erro parecer ser um problema de nível de plataforma (não causado pela sua configuração), registre o código de erro e contate o suporte Anaplan. Não gaste tempo com erros de plataforma durante um projeto de entrega.

A regra dos 90%

ℹ Espere erros, entregue sistematicamente

Cada configuração de cliente vai produzir um conjunto diferente de erros. O objetivo não é zero erros antes de prosseguir — é ir de 0% a 90% rapidamente. Categorize erros, corrija os resolvíveis de forma sistemática e registre os restantes como uma lista de issues conhecidos. "Os últimos 10% são o que você trabalha metodicamente" — não é sinal de que o projeto está quebrado.

Exercício de Error Log

1
Exercício de Error Log
⏱ 30 min
Objetivo Encontre todos os erros no modelo gerado. Categorize cada um por tipo de erro. Identifique a causa raiz de ao menos um erro LINE_ITEM_FORMULA e proponha uma correção.
  1. 1No Application Framework, abra o Error Log. Exporte para Excel se tiver mais de 10 erros.
  2. 2Crie uma tabela simples: Código do Erro | Tipo do Erro | Modelo | Causa Raiz | Ação
  3. 3Para cada erro, categorize como: LINE_ITEM_FORMULA, LIST_PROPERTY_FORMULA, IMPORT_ACTION ou MODEL_DATA_SOURCE.
  4. 4Escolha o primeiro erro LINE_ITEM_FORMULA. Anote o código (prefixo underscore).
  5. 5Vá para o modelo afetado. Procure o código em Line Items. Confirme que encontrou o Line Item correto.
  6. 6Abra o Line Item em blueprint. Registre a fórmula atual (quebrada).
  7. 7Abra o modelo base. Encontre o mesmo Line Item. Registre a fórmula do modelo base.
  8. 8Compare as duas fórmulas. Identifique qual referência mudou (renomeação de Hierarquia? geografia? Dimensão?).
  9. 9Corrija a fórmula no seu modelo OU copie-ajuste do modelo base. Confirme que resolve.
  10. 10Volte ao Error Log. Quantos erros restam? Quais erros são seguros para deixar por enquanto?
✅ Como deve ficar quando concluído

Perguntas de debrief

  1. Quais erros são seguros para deixar por enquanto? Quais precisam ser corrigidos antes do teste?
  2. Se um cliente perguntar "o modelo está pronto para usar?" e você tem 8 erros no log — o que você responde?
  3. Quando você escala um erro para o suporte Anaplan vs. corrige por conta própria?
  4. Como você documentaria sua abordagem de resolução de erros para o próximo consultor neste projeto?
