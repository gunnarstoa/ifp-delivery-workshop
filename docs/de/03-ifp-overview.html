Was IFP ist

Integrated Financial Planning (IFP) ist die vorgefertigte Applikation von Anaplan, die grundlegende FP&A-Anwendungsfälle für Planung, Budgetierung und Forecasting ermöglicht — für Umsatz, Betriebskosten, Investitionsausgaben, integrierte Finanzberichte und mehr — und unterstützt sichere Entscheidungsfindung sowie die fortlaufende Überwachung der finanziellen Gesundheit. Sie bietet einen strukturierten, vom Application Framework Configurator gesteuerten Ansatz zur Finanzplanung, der monatelange Modellbauzeit eliminiert und Partner-Beratern ein wiederholbares Delivery-Framework an die Hand gibt.

Version 2.1 ist das aktuelle Release. Sie baut auf der 2.0-Neuarchitektur auf, die das Application Framework eingeführt hat — ein vom Application Framework Configurator gesteuertes Modellgenerierungssystem, das die manuelle Modelleinrichtung ersetzt, zusammen mit der Integration des Anaplan Data Orchestrator (ADO) und Headcount Planning 2.0. Version 2.1 verfeinert die ADO-Konfiguration und erhöht die Dimensionalitätsgrenzen.

ℹ Ebenenbereiche

Der Standard sind 3 Ebenen pro Dimension, mit Ausnahme von Job (Standard: 4). Jede Hierarchie kann auf ein Minimum von 2 Ebenen reduziert oder auf maximal 15 Ebenen pro Dimension erhöht werden. Beim Entfernen von Ebenen muss die ursprüngliche Leaf-Ebene immer unten bleiben. Beim Hinzufügen wird die neue Ebene unten ergänzt und muss über die ursprüngliche Leaf-Ebene gezogen werden.

Provisionierungs-Voraussetzungen

Bevor eine Implementierung beginnen kann, müssen die folgenden Provisionierungsschritte vom Anaplan-Team durchgeführt werden:

  1. Kunde erwirbt die IFP-Applikation (inklusive Abonnementgebühr, die Upgrades abdeckt)
  2. Der Customer Success Business Partner öffnet ein Provisionierungsticket zur Erstellung des Workspace (mindestens 100 GB)
  3. Für ALM-Umgebungen (Dev/QA/Prod) werden separate Workspace-Provisionierungstickets für jede Umgebung benötigt
  4. Das Anaplan-Team stellt das Application Framework im Kunden-Tenant bereit
  5. Partner und Customer Success Business Partner werden informiert, wenn die Bereitstellung abgeschlossen ist
⚠ Tenant-Admin-Anforderung für die Generierung

Das Application Framework kann nur aus Ihrem Standard-Tenant generieren. Ein Partner, der seine Firmen-E-Mail verwendet (z. B. john@partnerfirm.com), kann nicht generieren, wenn der Kunden-Tenant nicht sein Standard ist. Lösung: Holen Sie sich eine E-Mail-Adresse in der Kundendomain oder lassen Sie den Tenant-Admin des Kunden generieren, während Sie ihn anleiten. Für die Generierung erforderliche Rollen: Application Owner, Workspace Admin, Page Builder, Integration Admin.

Modul-Architektur

IFPv2 erzeugt eine konsistente Modul-Namenskonvention über alle vier Modelle hinweg. Das Verständnis dieser Architektur ist essenziell für die Fehlerbehebung im Error Log und für Erweiterungsarbeiten.

PräfixModul-TypZweck
ADMINAdministrationKonfigurationseinstellungen, Mapping der Planungsmethoden, Benutzerzugriff
SYSSystemModellübergreifende System-Listen, Hierarchie-Mappings, Boolean-Steuerungen
DATDataStaging der Ist-Daten, Integrationen aus ADO und Headcount-Modell
INPInputPlanungs-Eingabe-Module — hier geben Anwender Forecast-Daten ein
CALCCalculationAggregation, Allokation, abgeleitete Berechnungen aus INP-Modulen
OUTOutputKonsolidierte Output-Module — speisen nachgelagerte Reports und Dashboards

Das Application-Framework-Konzept

Das Application Framework ist die Konfigurations- und Generierungs-Engine. Der Workflow ist:

1

Konfigurieren

Beantworten Sie alle Konfigurationsfragen im Application Framework Configurator. Diese definieren Hierarchie-Ebenen, Dimensionen, Planungsmethoden und Scope.

2

Generieren

Generierung starten. Das Application Framework liest Ihre Antworten und provisioniert alle vier IFP-Modelle (Financial Planning, Headcount, Admin, Data Orchestrator). Dies dauert 10–20 Minuten.

3

Überprüfen

Untersuchen Sie das generierte Modell und die Error Logs. „Generated with errors" ist normal und erwartet — es bedeutet nicht, dass das Modell defekt ist.

4

Erweitern

Fügen Sie kundenspezifische Erweiterungen auf das generierte Basismodell hinzu. Dokumentieren Sie alles, was Sie ändern.

🚨 „Generated with Errors" ist normal

Jede IFP-2.1-Generierung in einer Trainings- oder frühen Umgebung erzeugt Fehler. Dies sind Formelfehler, die durch Konfigurationsentscheidungen, Umbenennung von Hierarchien oder Geografie-Unterschiede zum Basis-Modell-Template verursacht werden. Das Modell ist funktionsfähig. Sie lernen im Abschnitt „Error Log Überprüfung", diese Fehler zu lesen und zu beheben.
