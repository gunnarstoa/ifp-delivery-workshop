Error Logs — Was sie sind

Error Logs werden vom Application Framework nach der Modellgenerierung erzeugt. Sie listen Formel- und Konfigurationsprobleme auf, die entstanden sind, weil sich das generierte Modell vom Basismodell-Template unterscheidet. Der Error Log ist unten rechts auf der Application Framework-Übersichtsseite zugänglich und kann nach Excel exportiert werden.

Error Logs sind neue Inhalte, die mit IFP 2.0 eingeführt wurden — frühere IFP-Versionen hatten dieses systematische Fehler-Reporting nicht. Sie effizient lesen zu lernen ist eine zentrale Praktiker-Kompetenz.

Referenz der Fehlertypen

FehlertypWas es bedeutetAktion
LINE_ITEM_FORMULAEine Formel referenziert ein Line Item, das im generierten Modell nicht existiert oder umbenannt wurde — oft aufgrund von Hierarchie-Umbenennungen oder KonfigurationsentscheidungenSuchen Sie das Line Item per Code (Unterstrich-Präfix), vergleichen Sie die Formel mit dem Basismodell, beheben Sie oder kopieren-anpassen Sie aus dem Basismodell
LIST_PROPERTY_FORMULAFormel referenziert eine Listeneigenschaft, die nicht existiert oder umbenannt wurdeSuchen Sie das Line Item per Code, vergleichen Sie die Formel mit dem Basismodell, aktualisieren Sie die Listeneigenschafts-Referenz
IMPORT_ACTIONEine Import-Action referenziert ein Quellmodul oder eine Liste, die sich geändert hatAktualisieren Sie das Import-Mapping im betroffenen Modell
MODEL_DATA_SOURCEEine Datenquellen-Verbindung ist defekt (ADO-Pipeline-Referenz)Stellen Sie die Verbindung in ADO wieder her oder aktualisieren Sie die Datenquellen-Referenz

Wie man einen Formelfehler findet

Der häufigste Fehlertyp ist LINE_ITEM_FORMULA. Hier ist der Diagnose-Workflow:

„Wir geben Ihnen in der Regel das Line Item mit einem Unterstrich am Anfang an — das ist der Fehler-Line-Item-Code. Gehen Sie ins Modell, suchen Sie nach diesem Code, und Sie landen direkt beim exakten Element."
— Workshop-Facilitator, Tag 1
1

Error Log öffnen

Klicken Sie aus der Application Framework-Übersicht auf das Error Log-Panel (unten rechts). Filtern Sie nur nach Formelfehlern. Bei mehr als 10 Fehlern exportieren Sie zuerst nach Excel.

2

Notieren Sie den Code

Jeder Fehler enthält einen Line Item-Code im Format _LINEITEMNAMEEXCEPTION (Unterstrich-Präfix). Dieser Code ist eindeutig — verwenden Sie ihn für die Modellsuche, nicht den Anzeigenamen.

3

Betroffenes Line Item finden

Gehen Sie zum betroffenen Modell (der Error Log sagt Ihnen, welches Modell). Öffnen Sie Line Items. Suchen Sie nach dem Code mit dem Unterstrich-Präfix. Das Suchergebnis bringt Sie direkt zum Modul und Line Item.

4

Mit Basismodell vergleichen

Öffnen Sie das Basismodell (Ihr Facilitator stellt den Zugang bereit). Suchen Sie dasselbe Line Item. Vergleichen Sie die generierte Formel mit der Formel des Basismodells. Identifizieren Sie, was sich geändert hat: Hierarchie-Umbenennung, Geografie-Entfernung, Dimensionsänderung.

5

Beheben oder kopieren-anpassen

Option A: Beheben Sie die Formel direkt — aktualisieren Sie Referenzen, sodass sie zu Ihrer Konfiguration passen. Option B: Kopieren Sie die Basismodell-Formel in Ihr Modell und passen Sie die Dimensionsreferenzen an. Option B ist für die meisten LINE_ITEM_FORMULA-Fehler sicherer und schneller.

6

Validieren

Prüfen Sie nach der Behebung, ob die Formel sich auflöst (kein roter Formelfehler-Indikator). Kehren Sie zum Error Log zurück und markieren Sie den Fehler als behoben. Fahren Sie mit dem nächsten Fehler fort.

7

Eskalieren, wenn Sie feststecken

Wenn der Fehler ein Problem auf Plattformebene zu sein scheint (nicht durch Ihre Konfiguration verursacht), protokollieren Sie den Fehlercode und wenden Sie sich an den Anaplan-Support. Verbringen Sie während eines Delivery-Engagements keine Zeit mit Plattformfehlern.

Die 90%-Regel

ℹ Erwarten Sie Fehler, liefern Sie systematisch

Jede Kundenkonfiguration erzeugt eine andere Fehlermenge. Das Ziel ist nicht null Fehler, bevor Sie fortfahren — es ist, schnell von 0 % auf 90 % zu kommen. Kategorisieren Sie Fehler, beheben Sie die lösbaren systematisch und erfassen Sie die verbleibenden als Liste bekannter Probleme. „Die letzten 10 % sind das, was Sie methodisch durcharbeiten" — kein Zeichen, dass das Projekt defekt ist.

Error Log Übung

1
Error Log-Übung
⏱ 30 Min.
Ziel Finden Sie alle Fehler im generierten Modell. Kategorisieren Sie jeden nach Fehlertyp. Identifizieren Sie die Ursache mindestens eines LINE_ITEM_FORMULA-Fehlers und schlagen Sie eine Behebung vor.
  1. 1Öffnen Sie aus dem Application Framework den Error Log. Exportieren Sie nach Excel, wenn Sie mehr als 10 Fehler haben.
  2. 2Erstellen Sie eine einfache Tabelle: Fehlercode | Fehlertyp | Modell | Ursache | Aktion
  3. 3Kategorisieren Sie jeden Fehler als: LINE_ITEM_FORMULA, LIST_PROPERTY_FORMULA, IMPORT_ACTION oder MODEL_DATA_SOURCE.
  4. 4Wählen Sie den ersten LINE_ITEM_FORMULA-Fehler. Notieren Sie den Code (Unterstrich-Präfix).
  5. 5Gehen Sie zum betroffenen Modell. Suchen Sie nach dem Code in Line Items. Bestätigen Sie, dass Sie das korrekte Line Item finden.
  6. 6Öffnen Sie das Line Item im Blueprint. Erfassen Sie die aktuelle (defekte) Formel.
  7. 7Öffnen Sie das Basismodell. Suchen Sie dasselbe Line Item. Erfassen Sie die Basismodell-Formel.
  8. 8Vergleichen Sie die beiden Formeln. Identifizieren Sie, welche Referenz sich geändert hat (Hierarchie-Umbenennung? Geografie? Dimension?).
  9. 9Beheben Sie die Formel in Ihrem Modell ODER kopieren-anpassen Sie aus dem Basismodell. Bestätigen Sie, dass sie sich auflöst.
  10. 10Kehren Sie zum Error Log zurück. Wie viele Fehler verbleiben? Welche Fehler können vorerst belassen werden?
✅ So sollte es nach Abschluss aussehen

Debriefing-Fragen

  1. Welche Fehler sind vorerst sicher zu belassen? Welche müssen vor dem Testen behoben werden?
  2. Wenn ein Kunde fragt „Ist das Modell einsatzbereit?" und Sie haben 8 Fehler im Log — was sagen Sie?
  3. Wann eskalieren Sie einen Fehler an den Anaplan-Support, anstatt ihn selbst zu beheben?
  4. Wie würden Sie Ihren Ansatz zur Fehlerbehebung für den nächsten Berater in diesem Projekt dokumentieren?
