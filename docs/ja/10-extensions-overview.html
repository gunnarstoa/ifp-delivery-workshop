拡張機能とは何か

拡張機能とは、Application Framework Configuratorでは生成できない、IFPベースモデルに対するお客様固有の追加または変更を指します。拡張機能は3つのカテゴリに分かれ、カテゴリによってリスクとアプローチが決まります。

拡張機能タイプ内容アップグレードリスク使用するタイミング
構成（Configure）Application Framework Configuratorを使用して、組み込み機能を有効化するなし — 公式にサポートされています常に最初に検討してください。Application Framework Configuratorに該当する設定があれば、Application Framework Configuratorを使用します。
拡張（Extend / Connect）ベースモデルの数式に手を加えずに、新しいラインアイテム・モジュール・リストを追加する低 — 新規追加した項目はアップグレードで上書きされませんApplication Framework Configuratorではカバーできないが、ベースモデルのロジックを変更せずに追加できる要件の場合。
変更（Amend / Modify / Delete）既存のベースモデルの数式・リスト・モジュールを変更する高 — アップグレードで変更が失われたり壊れたりする可能性があります最終手段。すべてのオーバーライドはData Tagでドキュメント化してください。外部ドキュメントも維持します。

IFP拡張機能のための7つのベストプラクティス

1

まず構成（Configure）から

Application Frameworkを通じた構成は、お客様の入力と要件に基づいてアプリケーションの構造を定義する主要な手段です。

2

次に拡張または接続（Extend / Connect）

拡張機能は常に「追加型」であるべきです。つまり、既存のロジックを変更せずに新しい機能を追加します。

3

変更（Modify）は慎重に

既存オブジェクト——特に構成不可能なオブジェクト——を変更すると、構成性とアップグレード性の両方に影響します。

4

削除（Remove）は注意して

構成済みアプリケーションからコンポーネントを削除すると、将来の再構成時や新しいアプリケーションバージョン採用時に追加の工数が発生する可能性があります。

5

一貫した命名規則を維持する

一貫した命名により、拡張機能の保守・サポート・アップグレードが容易になります。

6

関連する拡張機能をまとめる

複数の拡張機能をグループ化することで、特にアップグレード時に識別と管理が容易になります。

7

すべての拡張機能をドキュメント化する

すべての拡張機能と変更は明確にドキュメント化されている必要があります。

Data Tag — 最初に作成してください

⚠ ビルドの前にタグ付けを

IFPモデルに拡張変更を加える前に、2つのData Tagを作成してください：Extension（追加するすべての新規ラインアイテム用）とFormula Override（変更するベースモデルの数式用）です。これらのタグはモデル内における変更履歴の唯一の記録となります。タグがないと、将来別のコンサルタントが変更内容を知る手段がなくなります。

⚠ Formula Overrideとアップグレード

ベースモデルのラインアイテムに対するFormula Overrideは、拡張機能のなかで最もリスクの高い操作です。AnaplanがIFPアップデートをリリースすると、Formula Overrideが失われたり壊れたりする可能性があります。常にFormula OverrideのData Tagを付け、外部でもドキュメント化してください。IFPアップグレード後は、Formula Overrideタグでフィルタリングし、一つひとつ再検証してください。
