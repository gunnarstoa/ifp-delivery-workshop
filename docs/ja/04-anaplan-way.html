6段階のIFPデリバリー方法論

IFPのデリバリーはAnaplanの標準的なApp デリバリー段階に従い、Application Framework Configuratorによる生成ワークフロー向けに適合されています。各段階を理解することで、お客様に対する適切な期待値設定ができ、どの活動がどこに属するかが分かります。

📚 参考ドキュメント

The Anaplan Way for Applications Methodology — 構造化されたアプリケーション実装のための完全な方法論。

1

Pre-Project

案件準備の段階です。成果物には以下が含まれます：

  • Session Zero — お客様との構成ディスカバリー
  • 構成ワークシートの完成（44の質問すべてに回答済み）
  • 階層の命名およびレベル数の決定
  • データ可用性の確認：実績データのソース、Headcountフィード、通貨設定
  • ワークスペースのプロビジョニング、デリバリーチームのワークスペース管理者アクセスの確認

2

Config

Application Framework Configuratorの段階です。デリバリーコンサルタントは：

  • Application Framework Configuratorにすべての構成回答を入力
  • トップレベルの質問、財務プランニング、Headcount、CapEx構成をレビュー
  • モデル生成をトリガー
  • Error Logをレビュー

3

Test & Deploy

クリーンな生成後の検証段階です：

  • リスト／階層を投入し、ADOパイプライン経由でサンプル実績データをロード
  • CALCモジュールの計算ラインアイテムをスポットチェック
  • UXページでプランニングメソッドを検証
  • お客様チームのキーユーザーとともにUAT

4

Extension Foundation

お客様固有の拡張機能を設計しスコープを定めます：

  • IFPベースモデルとお客様要件のギャップを特定
  • 各ギャップを分類：構成 vs. 拡張 vs. 変更
  • 各拡張機能のサイズを見積もり（S／M／L／XL）
  • モデル内に拡張機能タグを作成（Extension + Formula Override）

5

Iterative Extension

優先度順に拡張機能を構築します：

  • 各拡張機能：設計 → 構築 → テスト → ドキュメント化
  • 追加または変更したすべてのラインアイテムにタグ付け
  • 外部の設計仕様書にドキュメント化（モデル内のメモのみで済ませない）
  • 各拡張機能の後にベースモデル機能のリグレッションテスト

6

Extension Test & Deploy

最終検証と引き継ぎ：

  • 全拡張機能を組み込んだ完全なUAT
  • 本番データボリュームでのパフォーマンステスト
  • 引き継ぎドキュメント：拡張機能ログ、Formula Overrideマップ、アップグレードチェックリスト
  • お客様の管理者向けトレーニング

ℹ IFP固有の段階に関する注記

Session Zeroは極めて重要です。カスタムモデル構築とは異なり、Session Zeroで下したIFP構成上の判断は、生成後に変更することが困難または不可能です。階層レベル、地理構造、エンティティ構成は、誰かがApplication Framework Configuratorに触れる前に確定させておかなければなりません。Session Zeroでディメンションを見落とすと、Test & Deploy段階で完全な再生成が必要になります。

2種類のお客様タイプ

🌱 Greenfield

  • 新規ワークスペース、既存のAnaplanモデルなし
  • IFPが最初のアプリケーション
  • 既存モデルとの連携の複雑さがない
  • ADOをゼロから完全にセットアップ
  • リスク： Session Zeroディスカバリーが唯一の制約——ここを確実に押さえる

🔗 既存ワークスペース

  • 稼働中の既存モデルと並行してIFPをデプロイ
  • ワークスペース管理者がリストの衝突を管理する必要
  • ADOが既存のインポートアクションと共存する必要があるかも
  • リスト命名が既存リストと衝突しないようにする
  • リスク： 生成が既存のワークスペース設定に影響する可能性
