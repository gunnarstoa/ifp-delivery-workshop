Application Framework Configuratorとは

Application Frameworkのコンフィギュレーターは、IFPに関するすべての構成上の意思決定の唯一の窓口です。3つのカテゴリ——トップレベル質問（23）、財務プランニング構成（17）、Headcount構成（3）——に加えて追加設定を含む、44の構成質問を提示します。これらの回答が、生成される内容——モジュール構造、リスト階層、数式ロジック、利用可能なプランニングメソッド——を決定します。

Application Framework Configuratorには専用のURLからアクセスします。ブラウザのAnaplanホームURL内の「home」を「applications」に置き換えてください（v2.1.0では、通常のワークスペースナビゲーションにApplicationsリンクがまだ表示されません）。たとえば、ホームのAnaplan URLが「https://us1a.app.anaplan.com/home」であれば、URLを「https://us1a.app.anaplan.com/applications」に変更します。一般提供（GA）に達した時点で、このリンクは標準ナビゲーション内に表示される予定です。

i ベストプラクティス：まずベースコピーを生成する

構成を変更する前に、まずデフォルト設定のままアプリケーションを生成してください。これによりベースコピー——すべての数式が正しく生成され、デフォルトの階層名とデフォルトディメンションがそのまま入ったモデル——が得られます。後で構成済みの生成で数式エラーが発生した場合、ベースコピーで正しい数式を確認してコピーできます。この生成にはわかりやすいラベルを付けてください（例：「Base - no changes」）。

⚠ 構成は生成時に確定する

いったん生成をトリガーすると、構造に関する多くの判断はモデルを再生成しない限り変更できません。階層レベル、エンティティ構造、地理構成は、生成前に確定させる必要があります。これらはSession Zeroで正しく決めてください——後で変更するのが最も難しい項目です。

Application Framework Configurator概要（v2.1）

構成セクション

階層

階層セクションは、生成されるモデルのディメンション構造を定義します。ここでの決定はすべてのモジュールに波及し、完全な再生成なしには生成後に変更できません。

  • Location： 何レベルにするか？（例：Global → Region → Country → City）どのレベルがプランニングレベルで、どのレベルがロールアップ専用か？
  • Functional Area： どのファンクショナルエリアがスコープに含まれるか？（Gross Profit、OpEx、Headcount、Balance Sheet）有効化された各エリアは追加のモジュールを生成します。
  • Entity： エンティティのレベル数は？最上位の連結エンティティは何か？
  • Product： 製品レベルのプランニングが必要か？製品階層のレベル数は？

トップレベルの質問は影響が最も大きい項目です。これらがIFPモデルの構造的な骨格を定義します。

ディメンション設定

ℹ Dimensionality質問が「Applies To」に与える影響

これらの質問（13〜17）は、モジュール生成時の「Applies To」に影響します。デフォルトでは、すべてのディメンションが使用される前提のため、オプションディメンションがすべて選択されています。不要と判断されたディメンションがあれば、オフにしてください。

ディメンションをオフにすると、生成されるモジュールの複雑性が下がり、不要なディメンションの組み合わせを計算から除外することでパフォーマンスが向上します。

Configuratorの主な留意点（デリバリーから得た学び）

  • 階層名とApplication Framework Configuratorの質問は一致させる必要があります： v2.1.0では、Hierarchiesセクションの階層名と、構成質問内のリネームは連動していません。両方を更新する必要があります。将来のリリースで修正予定です。
  • 名前の末尾に空白を入れない： リネームした階層の末尾の空白が生成エラーの原因となった事例があります。名前の前後に空白がないかダブルチェックしてください。
  • リーフレベルは最下部に維持する： レベルを削除する場合は中間レベルを削除し、リーフを最下部に残します。レベルを追加すると、システムは最下部に追加するため、元のリーフレベルの上にドラッグしてください。元のリーフを新しいレベルと同じ名前にリネームすることはできません。
  • HierarchiesとConfiguration Questionsは別物： リネームしたディメンションごとに2つの質問があります——一つはリストを作成、もう一つはモジュールを作成します。後者は小文字です。両方を更新する必要があります。
  • 階層ごとのRestore Default： 各階層に固有の「Restore Default」ボタンがあります。
  • Configuratorと階層の順序： 階層から先に進めても、構成質問から先に進めても構いません——決まった順序はありません。
  • 今すぐ必要なくてもCapExを生成する： Polarisモジュールはデータが入るまで大きなメモリを使いません。お客様が後で追加した場合の再生成を避けるため、今のうちにCapExを生成しておきましょう。
  • カスタムディメンション（フラットリスト）は拡張機能であり、階層ではない： フラットリスト形式のディメンション（例：Channel）はHierarchiesセクションで構成するものではなく、生成後に拡張機能として構築します。

生成プロセス

すべての構成質問と階層設定を完了したら：

  1. Application Frameworkの概要からGenerateをクリック
  2. この生成の説明を入力（例：「Base — no changes」または「Customer Name — 3 entity levels, no geography」）
  3. ターゲットワークスペースを選択
  4. 生成はクラウド上で実行され、約20〜40分かかります。進捗はApplication Framework内ではなく、メインの/applicationsページで監視してください（フレームワーク内のステータス表示は遅延する場合があります）
  5. 完了すると、すべての項目が緑色で表示されます。赤はエラー、黄色は警告です。
  6. Error Logをレビューします。詳細は「Error Logレビュー」のセクションを参照してください。
