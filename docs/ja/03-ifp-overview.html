IFPとは

Integrated Financial Planning（IFP）はAnaplanの事前構築済みアプリケーションであり、収益、営業費用、資本支出、統合財務諸表などを対象としたプランニング、予算策定、フォーキャストといったFP&Aの基幹ユースケースを実現します。これにより、確信を持った意思決定と継続的な財務健全性のモニタリングを支援します。Application Framework Configuratorによって駆動される構造化された財務プランニング手法を提供し、何ヶ月にも及ぶモデル構築時間を不要にし、パートナーコンサルタントに再現性のあるデリバリーフレームワークをもたらします。

バージョン2.1が現行リリースです。これは、手動でのモデルセットアップを置き換えるApplication Framework Configurator駆動のモデル生成システムであるApplication Frameworkと、Anaplan Data Orchestrator（ADO）連携、およびHeadcount Planning 2.0を導入した2.0の再設計を基盤としています。バージョン2.1ではADO構成が洗練され、ディメンションの上限が拡張されました。

デフォルトは各ディメンションにつき3レベルですが、Jobのみデフォルトが4レベルです。各階層は最小2レベルまで減らすことができ、最大15レベルまで増やせます。レベルを削除する場合、元のリーフレベルは常に最下部に残さなければなりません。レベルを追加する場合、新しいレベルは最下部に追加されるため、元のリーフの上にドラッグする必要があります。

プロビジョニング要件

実装を開始する前に、Anaplanチームが以下のプロビジョニング手順を完了する必要があります：

  1. お客様がIFPアプリケーションを購入（サブスクリプション料金にはアップグレードが含まれます）
  2. Customer Success Business Partnerが、ワークスペース作成（最小100GB）のプロビジョニングチケットを起票
  3. ALM環境（dev／QA／prod）の場合、環境ごとに個別のワークスペース プロビジョニングチケットが必要
  4. AnaplanチームがApplication Frameworkをお客様のテナントへデプロイ
  5. デプロイ完了時にパートナーとCustomer Success Business Partnerに通知
⚠ 生成にあたってのテナント管理者要件

Application Frameworkは、デフォルトテナントからしか生成できません。自社のメール（例：john@partnerfirm.com）を使うパートナーは、お客様のテナントがデフォルトでない場合は生成できません。解決策：お客様のドメイン内のメールアドレスを取得するか、お客様のテナント管理者に手順を案内しながら生成してもらってください。生成に必要なロール：Application Owner、Workspace Admin、Page Builder、Integration Admin。

モジュールアーキテクチャ

IFPv2は、4つのモデル全体で一貫したモジュール命名規約に従って生成されます。このアーキテクチャを理解することは、Error Logのトラブルシューティングや拡張機能の構築に不可欠です。

プレフィックスモジュール種別目的
ADMINAdministration構成設定、プランニングメソッドのマッピング、ユーザーアクセス
SYSSystemモデル横断のシステムリスト、階層マッピング、Boolean制御
DATData実績データのステージング、ADOおよびHeadcountモデルからの連携
INPInputプランニング入力モジュール——ユーザーがフォーキャストデータを入力する場所
CALCCalculationINPモジュールからの集計、配賦、派生計算
OUTOutput統合された出力モジュール——下流のレポートとダッシュボードへ連携

Application Frameworkの考え方

Application Frameworkは構成および生成のエンジンです。ワークフローは以下のとおりです：

1

構成

Application Framework Configuratorですべての構成質問に回答します。これにより、階層レベル、ディメンション、プランニングメソッド、スコープが定義されます。

2

生成

生成をトリガーします。Application Frameworkが回答内容を読み取り、IFPの4つのモデル（Financial Planning、Headcount、Admin、CapEx）すべてをプロビジョニングします。所要時間は20〜40分です。

3

レビュー

生成されたモデルとError Logを確認します。「Generated with errors」は正常で想定どおりの状態であり——モデルが壊れているという意味ではありません。

4

拡張

生成されたベースモデルの上に、お客様固有の拡張機能を追加します。変更点はすべて文書化してください。

🚨 「Generated with Errors」は正常です

トレーニングや初期段階の環境におけるIFP 2.1の生成は、必ずエラーを生み出します。これらは構成上の選択、階層のリネーム、ベースモデルテンプレートとの地理的差異に起因する数式エラーです。モデル自体は機能しています。これらのエラーの読み方と解決方法は、Error Logレビューのセクションで習得します。
