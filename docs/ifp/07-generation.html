Model Generation

Trigger generation, watch the four IFP models build out, and read what the status tells you

Configuration⏱ 30 minGuided practice

What you'll do

  • Trigger generation from the Application Framework Configurator
  • Recognize what the four status states mean, especially Generated with Errors
  • Verify all four expected models exist in the workspace and capture a baseline error count
  • Internalize that "errors after generation" is normal — they're fixed in Error Log Review, not by re-generating

What generation actually does

When you click Generate, the Application Framework reads all 44 of your configuration answers and provisions a complete IFP model structure specific to those answers. This is dynamic provisioning, not a template copy. Generation produces four models:

  • Financial Planning — the main planning model (ADMIN, SYS, DAT, INP, CALC, OUT modules)
  • Headcount — workforce planning with Job-level detail (when enabled)
  • Admin — the mapping hub for source-to-planning metadata (accounts, entity, department)
  • CapEx — capital expenditure planning (only when enabled)

It also creates the UX app participants will navigate, plus ~17 assets (import definitions, saved views, dashboard configurations).

The four generation status states

StatusWhat it meansWhat to do
Generating…Provisioning is in progress.Wait. Don't trigger another generation. Expected: 20–40 min.
Generated with ErrorsGeneration completed but some formulas reference items that don't match the base template (usually because of renames or removed dimensions).Normal and expected. The model works. Errors are resolved in Error Log Review.
Generated SuccessfullyGeneration completed with zero formula errors.Proceed straight to Post-Generation Steps.
Generation FailedA platform error stopped generation.Don't retry blindly. Contact Anaplan support so the failure is diagnosed first.
🚨 "Generated with Errors" is normal — don't re-generate

This status almost always means a formula referenced a dimension or item that you renamed or turned off in the Configurator. The model is functional and you'll work through the error list systematically in Error Log Review. Re-running generation won't change anything — the errors are baked into the configuration, not into the generation pipeline.

Step 1 — Trigger generation

You only do this once per exercise. From here on, the Application Framework does the work.

  1. From the Application Framework Configurator, confirm every section is answered — every "DEFAULT ANSWER" or "CUSTOM ANSWER" indicator should be settled the way you intended.
  2. Click Generate.
  3. Confirm any prompts that appear.
  4. Note the start time. Generation typically takes 20–40 minutes.

Step 2 — Monitor status while generation runs

The Application Framework overview page is the source of truth for status. Don't close the browser, don't trigger another generation, don't refresh the workspace looking for the models — they appear together when generation finishes.

  1. Stay on the Application Framework overview page (or come back to it periodically).
  2. Refresh the page every few minutes to read the current status.
  3. If the status flips to Generation Failed, stop and capture a screenshot for support — don't restart on your own.

Step 3 — Verify the four models in your workspace

Once the status reads Generated Successfully or Generated with Errors, the four models exist in your workspace.

  1. From the workspace, open the Models list. Confirm four models are present: Financial Planning, Headcount, Admin, and CapEx (CapEx is only there if the scenario enabled it).
  2. Open the Financial Planning model.
  3. Navigate to Modules. Confirm modules with each of these prefixes exist: ADMIN, SYS, DAT, INP, CALC, OUT.
  4. Spot-check that the planning sub-processes match the scenario — Revenue & COGS, Operating Expense, Headcount, CapEx Planning, Balance Sheet & Cash Flow should appear when the scenario enabled them, and be absent when it didn't.

Step 4 — Record your baseline error count

The error log is the bridge to the next page. Capturing the count now gives you a number to compare against as you work through resolutions.

  1. Return to the Application Framework overview page.
  2. Find the error log indicator in the bottom-right of the page.
  3. Note the total error count. Record it in your notes — this is your baseline for Error Log Review.
  4. Click into the error log briefly to see the first few entries. You don't need to resolve them here; that's what the next two pages are for.
End-of-step screenshot to capture Application Framework overview page showing the final status (Generated Successfully or Generated with Errors), the four models listed, and the error count indicator with your baseline number visible.

Once the four models exist in your workspace and you've recorded the baseline error count, run the check below.

Generation results look good.

  • Status → Generated Successfully or Generated with Errors
  • Models in workspace → Financial Planning, Headcount, Admin (plus CapEx if enabled)
  • Module prefixes → ADMIN, SYS, DAT, INP, CALC, OUT all present in Financial Planning
  • Baseline error count → recorded in your notes

Your generation produced the expected structure. Move on to Post-Generation Steps to verify the model and then to Error Log Review.

You're done with generation

Your model exists. Next up is Post-Generation Steps — a short verification checklist to confirm the model is in a stable shape before you spend any time on data or extensions. After that, Error Log Review walks through working down your baseline error count.
