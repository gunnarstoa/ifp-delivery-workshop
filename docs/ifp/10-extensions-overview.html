Extensions Overview

The three extension types, 7 best practices, and the two Day 2 labs

30 minWalkthrough

What Extensions Are

Extensions are customer-specific additions or modifications to the IFP base model that the Application Framework Configurator cannot produce. They fall into three categories — and the category determines the risk and the approach.

Extension TypeWhat It IsUpgrade RiskWhen to Use
ConfigureUse the Application Framework Configurator to enable a built-in capabilityNone — this is supportedAlways try this first. If the Application Framework Configurator has a setting for it, use the Application Framework Configurator.
Extend / ConnectAdd new line items, modules, or lists without touching base model formulasLow — new items are not overwritten by upgradesWhen the customer needs something the Application Framework Configurator doesn't cover, but you can add it without changing base model logic.
Amend / Modify / DeleteChange an existing base model formula, list, or moduleHigh — changes may be lost or broken on upgradeLast resort. Document every override with a data tag. Maintain external documentation.

7 Best Practices for IFP Extensions

1

Configure First

Configuration through the Application Framework is the primary way to define an application's structure based on customer inputs and requirements.

2

Extend or Connect Next

Extensions should always be additive, meaning they add new capability without changing existing logic.

3

Modify with Care

Modifying existing objects — especially those that cannot be configured — can impact both configurability and upgradability.

4

Remove with Caution

Removing components from a post-configured application can create additional effort during future reconfigurations or when adopting a new application version.

5

Maintain a Consistent Naming Convention

Consistent naming makes extensions easier to maintain, support, & upgrade.

6

Group Related Extensions Together

Grouping multiple extensions makes them easier to identify and manage, especially during upgrades.

7

Document all Extensions

All extensions & modifications should be clearly documented.

Data Tags — Create These First

⚠ Tag Before You Build

Before making any extension changes to an IFP model, create two data tags: Extension (for all new line items you add) and Formula Override (for any base model formula you change). These tags are the only in-model record of what you've changed. Without them, a future consultant has no way to know.

⚠ Formula Overrides and Upgrades

Formula overrides on base model line items are the highest-risk extension action. When Anaplan releases an IFP update, formula overrides may be lost or broken. Always tag them with the Formula Override data tag and document them externally. After any IFP upgrade, filter by the Formula Override tag and re-validate each one.
