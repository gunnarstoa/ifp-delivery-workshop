Post-Generation Steps

The verification checklist to run after every generation before any other work

20 minWalkthrough
🔍 Post-Generation Verification — Workshop vs. Reality

This page covers the essential high-priority checks for a working model.

In a real production implementation, post-generation validation is much more extensive — typically 1–2 days of systematic review covering:

  • Configuration verification (hierarchy structure, dimension settings)
  • Security and access validation
  • Performance optimization and formula efficiency
  • Documentation and handoff prep
  • Data load validation (actuals integration)
  • User acceptance testing scenarios

The complete post-generation checklist is in Section 5 and Appendix A of the Configuration Guide.

This workshop focuses on core concepts and workflow. For live implementations, always follow the complete Configuration Guide checklist.

Required After Every Generation

After generation completes, run through the following checklist before doing any other work. This ensures you have a complete, stable baseline before adding data or extensions.

  • Verify all 4 models exist in the workspace (Financial Planning, Headcount, Admin, CapEx)
  • Check application objects count — Application Framework overview shows: models, UX apps, assets, data orchestrator objects
  • Open error logs and note total error count — record this as your baseline
  • Review the Error Log to verify all modules were generated correctly — any missing modules will surface here as errors. See the Error Log Review section for details.
  • Verify key SYS module line items are populated: Planning Methods list will come pre-populated. All production data lists (entity, department, accounts) will come unpopulated and are populated by pushing data from Data Orchestrator.
  • Confirm hierarchy lists loaded with expected members. All hierarchy list members are populated via ADO — some via CSV or connected external source. Hierarchies like entity, department, IS Accounts, and BS Accounts require an additional mapping step in the Admin Model to map source list members to their planning equivalents before pushing to other spoke models.
  • Check that Planning Method list members are populated — this is the only item verifiable at this stage. The SYS and ADMIN modules tied to Planning Methods are populated afterwards; refer to Appendix A3 of the Configuration Guide to complete this step.
  • Run a spot-check on a calculated line item in a CALC module — confirm formula is not blank
  • ▶ Watch: IFPv2 Models — Post Generation walkthrough — short video covering the post-generation verification steps above. (Opens in SharePoint · Anaplan sign-in required)
⚠ Blank Formulas? Check the Error Log First

The #1 way to verify if a formula should be there and turns out blank is reviewing the Error Log. All Formula Errors will cause the formula to be BLANK — they fail due to an invalid reference, either from line item name changes not recognized or structure changes. Always check the Error Log before assuming a formula is missing.

Post-Generation Mini-Lab

1
Verify Your Generated Model
⏱ 20 min
Objective Work through the post-generation checklist. Identify the baseline error count and confirm the model structure is complete.
  1. 1From your workspace, confirm 4 models are listed. Name them in your notes.
  2. 2Open the Application Framework overview page. Record: model count, UX app count, asset count, error count.
  3. 3Click on Error Logs (bottom-right). Record the total error count — this is your baseline.
  4. 4Open the Financial Planning model. Navigate to Modules. Verify at least one module exists for each prefix: ADMIN, SYS, DAT, INP, CALC, OUT.
  5. 5Open a SYS module. Find the Planning Methods list. Verify the methods you enabled in the Application Framework Configurator are present.
  6. 6Open a CALC module (e.g., CALC OPX IS Itemized or similar). Open a line item in blueprint. Verify the formula field is not blank.
  7. 7Mark your checklist complete. You are ready to review error logs.
