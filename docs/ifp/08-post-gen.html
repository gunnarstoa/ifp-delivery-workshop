Post-Generation Steps

The verification checklist to run after every generation before any other work

Configuration⏱ 20 minGuided practice

What you'll do

  • Walk through a short verification checklist on the model the Application Framework just generated
  • Confirm all four models exist and the Financial Planning module structure is complete (ADMIN, SYS, DAT, INP, CALC, OUT)
  • Capture and record the baseline error count from the Application Framework overview
  • Recognize the difference between an unpopulated production list (expected) and a broken formula (needs Error Log Review)
🔍 Workshop vs. real-world verification

This page covers the essential high-priority checks that confirm the model is structurally complete. In a real production implementation, post-generation validation is typically 1–2 days of systematic review across configuration verification, security and access, performance, documentation, data-load validation, and user-acceptance scenarios.

The complete checklist lives in Section 5 and Appendix A of the Configuration Guide. For live implementations, always follow that checklist end-to-end — what you'll do below is a useful subset of it, not a replacement.

Before you start: what you can verify now vs. later

Two categories of "empty-looking" content are easy to confuse right after generation:

  • Verifiable now — module structure (ADMIN/SYS/DAT/INP/CALC/OUT modules exist), the four models in the workspace, the Planning Methods list pre-populated in SYS modules, and formulas non-blank in CALC modules.
  • Populated later — production data lists like Entity, Department, Accounts. These ship empty by design and are populated by pushing data from the Data Orchestrator. Several hierarchy lists also need a mapping step in the Admin model before they flow to spoke models.

Step 1 — Confirm the four models exist in your workspace

This is the simplest check and the fastest way to know generation didn't drop a model.

  1. Open the workspace and view the Models list.
  2. Confirm Financial Planning, Headcount, and Admin are all listed.
  3. If the scenario enabled it, confirm CapEx is also listed.
  4. Note any model that's missing — a missing model is something the error log will explain.

Step 2 — Record the application-framework counts

The overview page summarizes the build into a handful of counts. Capturing them now gives you the baseline to compare against any future re-generation or extension work.

  1. Open the Application Framework overview page.
  2. Record four counts in your notes:
    • Models
    • UX apps
    • Assets
    • Data orchestrator objects
  3. The expected shape: 4 models, 1+ UX app, ~17 assets. Note any wild deviation — large under-counts can indicate a partial generation.

Step 3 — Capture the baseline error count

This number is your reference point for the next page. Resolving errors is incremental — start by knowing where you began.

  1. From the Application Framework overview page, find the Error Logs indicator in the bottom-right.
  2. Record the total error count. Label it "baseline" in your notes.
  3. Click into the log briefly and skim the first few entries to get a feel for the kinds of errors — most will be LINE_ITEM_FORMULA references that don't match a renamed dimension. You'll work them in Error Log Review.

Step 4 — Verify Financial Planning module structure

A generated Financial Planning model has modules across six prefixes. Missing prefix usually means missing functionality, and that surfaces in the error log too.

  1. Open the Financial Planning model.
  2. Navigate to Modules.
  3. Confirm at least one module exists for each prefix: ADMIN, SYS, DAT, INP, CALC, OUT.
  4. If a prefix is completely missing, note it in your error log review queue — it's usually traced to a top-level question or a dimension setting in the Configurator.

Step 5 — Spot-check the Planning Methods list

The Planning Methods list is the only data-shaped thing that should be populated right after generation. Lists like Entity, Department, and Accounts are populated later via Data Orchestrator.

  1. From the Financial Planning model, open a SYS module — for example, SYS by Planning Methods.
  2. Find the Planning Methods list.
  3. Confirm the planning methods you enabled in the Application Framework Configurator are present.
  4. Don't worry about empty production lists (Entity, Department, Accounts) — those load later, after the Admin model's mapping step pushes data to spoke models.

Step 6 — Spot-check a CALC formula

This is the only verification that catches a structural formula failure that doesn't show up in the error log. Blank formulas in CALC modules are caused by invalid references — and those references show up in the error log too, but the formula field being blank is the most visible symptom.

  1. Open a CALC module — for example, CALC OPX IS Itemized.
  2. Open a line item in blueprint.
  3. Confirm the Formula field is not blank.
  4. If a formula is blank, that's a Formula Error — go straight to the Error Log to find the failed reference.
⚠ Blank formulas? Check the error log first

Every Formula Error causes the formula field to render as BLANK. The cause is almost always an invalid reference — usually a renamed line item, a removed dimension, or a structural change from the base template. Don't assume the formula is "missing" — assume the reference is broken and go look in the error log.

End-of-step screenshot to capture Application Framework overview with the four counts visible, the Financial Planning model open showing module list with all six prefixes (ADMIN, SYS, DAT, INP, CALC, OUT), and a CALC module line item in blueprint with a non-blank Formula field.

Once all six checks above are done and your baseline error count is recorded, run the check below.

Post-generation verification complete.

  • Models present → Financial Planning, Headcount, Admin (plus CapEx if enabled)
  • Framework counts → models, UX apps, assets, data-orchestrator objects all recorded
  • Baseline error count → captured
  • Module prefixes → ADMIN, SYS, DAT, INP, CALC, OUT all present in Financial Planning
  • Planning Methods list → populated with the methods enabled in the Configurator
  • CALC formula spot-check → non-blank

The model is structurally complete and your baseline is recorded. Move on to Error Log Review to start resolving the errors you captured here.
