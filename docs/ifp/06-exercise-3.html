Exercise 3: Nuclear Plant — OpEx and Headcount with WBS

Configure IFP for a utilities company with deep cost-center attributes and a Work Breakdown Structure

Configuration⏱ About 30 minsGuided practice

What you'll do

  • Apply the standardized naming convention with the Exercise 3 date so this run is unique in the tenant
  • Configure the eight standard dimensions with four renames (Accounts → GL Accounts, Department → CostCenter, Vendor → Supplier, Functional Area → WBS)
  • Add four flat custom lists for downstream PO and management tracking
  • Attach four attributes to the CostCenter dimension so cost centers can be sliced by manager, entity, profit center, and country
  • Answer the top-level and Financial Planning questions for an OpEx-only scope

Reset the Configurator before you start

This exercise is net-new and independent of Exercise 2. Reset everything so prior answers don't carry over.

⚠ Complete all three resets in this order
  1. Restore default for the Configuration Questions — in the Configuration Questions section, click Restore default. Every gray "CUSTOM ANSWER" indicator should flip back to "DEFAULT ANSWER".
  2. Restore default for each Hierarchy — hierarchies must be restored one at a time. Open each one (Entity, Account, Location, etc.) and click Restore default individually. Confirm levels return to documented defaults.
  3. Clear Dataspace objects — remove any objects in your assigned Dataspace from the prior exercise so this generation starts from a clean slate.

The scenario

A utilities company operates a nuclear facility and needs operating expense and headcount planning with a detailed Work Breakdown Structure (WBS) for project hierarchy. They do not need balance sheet, cashflow, margin planning, or CapEx in the base model — CapEx project and purchase-order tracking are added in the extension phase. The model uses a rich set of attributes on the CostCenter dimension so cost centers can be sliced by manager, entity, profit center, and country.

Before you start: the six rules

The same rules from Exercise 1 still apply — refer back as you work.

⚠ Rules to follow throughout the exercise
  • Use a standardized naming convention for every model and app — duplicate names in the tenant cause generation to fail
  • Renames must be no longer than the original dimension name, especially "Department"
  • All standard dimensions are composite hierarchies. Flat custom lists are handled in the extension layer
  • Minimum 2 levels per dimension, maximum 15
  • When modifying hierarchy levels, add or remove from the middle — never from the leaf
  • Click Apply Changes after every modification. If you need to start over, use Restore default

The scenario says CapEx isn't in scope for this exercise. Why still rename the capex model?

Correct. The renames happen at the model-management level, not at generation. Every base model in the tenant is renamed so each exercise leaves the tenant in a unique, attributable state — even models the current scenario won't actually exercise.

Step 2 — Configure the eight standard dimensions

This scenario has four renames — including Functional Area repurposed as the Work Breakdown Structure (WBS) with 5 levels for the project hierarchy. Click Apply Changes after each dimension before moving on.

  1. In the Configurator, open the Dimensions section.
  2. For Accounts, keep it required and rename it to GL Accounts. Click Apply Changes.
  3. For Entity, keep it required, leave the name as-is, and set # of levels to 2. Click Apply Changes.
  4. For Department, keep it required, rename it to CostCenter, and set # of levels to 3. Click Apply Changes.
  5. For Job, keep it required, leave the name as-is, and set # of levels to 2. Click Apply Changes.
  6. For Vendor, keep it required, rename it to Supplier, and set # of levels to 2. Click Apply Changes.
  7. For Functional Area, keep it required, rename it to WBS, and set # of levels to 5 for the Work Breakdown Structure. Click Apply Changes.
  8. For Customer, mark it not required. Click Apply Changes.
  9. For Location, mark it not required. Click Apply Changes.

Quick check

This scenario repurposes Functional Area as WBS with 5 levels. What's the reason for picking Functional Area for that?

Correct. Repurposing is a common pattern. A standard dimension the scenario doesn't need for its original meaning — like Functional Area here — becomes the slot you reuse for something that needs the composite-hierarchy shape, such as a 5-level WBS.
Not quite. The right answer is B. Repurposing reuses an unused optional dimension as a composite hierarchy. There's no hard rule forcing this — Functional Area is just convenient here because the utility scenario doesn't need it for its original meaning.

Step 3 — Add four flat custom lists

The utilities scenario tracks countries, PO headers, PO items, and managers as flat lists. Three of them (PO Headers, PO Items, Managers) are placeholders for the CapEx and PO extension exercises, but they're declared here so the extension layer can reference them.

  1. Open the Custom Lists section in the Configurator.
  2. Add Country as a Flat custom list — supporting attribute list for Cost Center attribution.
  3. Add PO Headers as a Flat custom list — placeholder for purchase-order tracking in the extension exercise.
  4. Add PO Items as a Flat custom list — placeholder for line-item detail in the extension exercise.
  5. Add Managers as a Flat custom list — used by the CostCenter manager attribute in Step 4.
  6. Click Apply Changes to save all four lists.

Quick check

The scenario says PO Headers and PO Items are part of the CapEx extension, not the base generation. Why declare them in the Configurator at all?

Correct. Extensions don't create custom lists from nothing — they reference lists that already exist in the tenant. Declaring the PO Headers / PO Items / Managers lists during the base configuration is what lets the extension exercise wire them into the model without manual setup.
Not quite. The right answer is C. The base configuration prepares the tenant for the extension by creating the custom lists the extension will reference by name.

Step 4 — Configure CostCenter dimension attributes

The utilities scenario uses a rich attribute structure on CostCenter so each cost center carries a manager, an entity, a profit center, and a country.

  1. Open the Dimensions section in the Configurator and expand CostCenter.
  2. Add an attribute named Cost Ctr Manager, type List, attribute list Managers. Click Apply Changes.
  3. Add an attribute named Entity, type List, attribute list E2 Entity. Click Apply Changes.
  4. Add an attribute named Profit Center, type List, attribute list C3 Cost Ctr. Click Apply Changes.
  5. Add an attribute named Country, type List, attribute list Country. Click Apply Changes.

Once all four CostCenter attributes are added, run the check below.

Quick check

Why use a dimension attribute for "Cost Ctr Manager" instead of adding Managers as its own dimension?

Correct. Adding Managers as a dimension would multiply every planning line item by every manager — a planning axis. As an attribute, each cost center just points to a single manager, and reports can group cost centers by their attached manager without inflating the model.
Not quite. The right answer is A. The decision is shaped by what you're planning against versus what you're slicing by. Managers is a slicer for cost centers — attributes are the right tool for that.

Step 5 — Answer the top-level and Financial Planning questions

This scenario narrows scope back to OpEx and Headcount only — Margin Planning is set to Expense Only and balance-sheet, cashflow, top-down, and allocations are all turned off.

  1. Open the Top-Level Questions section in the Configurator.
  2. Set Multi-currency support? to Yes.
  3. Set Use the Functional Area dimension? to Yes — the scenario repurposes it as WBS, but the dimension itself is still in scope.
  4. Click Apply Changes.
  5. Open the Financial Planning section.
  6. Set Top-down targets? to No.
  7. Set Margin Planning? to Expense Only — OpEx-only scope, no revenue or balance-sheet planning.
  8. Set Balance Sheet? to No.
  9. Set Cashflow reporting? to No.
  10. Set Expense allocations? to No.
  11. Click Apply Changes to save the Financial Planning answers.

Quick check

This scenario sets Balance Sheet and Cashflow reporting to No even though the utility's accountants do produce financial statements. What's the right reason?

Correct. Scope decisions come from the client's planning needs, not from technical constraints. The utility plans OpEx and headcount in IFP; their statements live elsewhere. Turning Balance Sheet and Cashflow off keeps the generated model focused on what's actually used.
Not quite. The right answer is B. There's no Configurator rule blocking these answers — they're switched off because financial-statement planning isn't part of the IFP scope for this client.
