Exercise 2: Insurance — 3-Statement Planning

Configure IFP for an insurance company that needs full P&L, Balance Sheet, and Cashflow

Configuration⏱ About 30 minsGuided practice

What you'll do

  • Apply the standardized naming convention with the Exercise 2 date so this run is unique in the tenant
  • Configure the eight standard dimensions for a 3-statement insurance scenario (Vendor renamed to Projects, Product deeply hierarchical)
  • Add the CapEx List custom dimension for capital project tracking
  • Answer the top-level and Financial Planning questions that widen the model to Balance Sheet and Cashflow

Reset the Configurator before you start

This exercise is net-new and independent of Exercise 1. Reset everything so prior answers don't carry over.

⚠ Complete all three resets in this order
  1. Restore default for the Configuration Questions — in the Configuration Questions section, click Restore default. Every gray "CUSTOM ANSWER" indicator should flip back to "DEFAULT ANSWER".
  2. Restore default for each Hierarchy — hierarchies must be restored one at a time. Open each one (Entity, Account, Location, etc.) and click Restore default individually. Confirm levels return to documented defaults.
  3. Clear Dataspace objects — remove any objects in your assigned Dataspace from the prior exercise so this generation starts from a clean slate.

The scenario

An insurance company needs comprehensive financial planning across the income statement, balance sheet, and cashflow. They run a multi-product portfolio with detailed cost-center and vendor tracking. CapEx project tracking will be added in the extension phase. You'll configure all three statements, deepen the Product hierarchy, and prepare the model for the CapEx extension.

Before you start: the six rules

The same rules from Exercise 1 still apply — refer back as you work.

⚠ Rules to follow throughout the exercise
  • Use a standardized naming convention for every model and app — duplicate names in the tenant cause generation to fail
  • Renames must be no longer than the original dimension name, especially "Department"
  • All standard dimensions are composite hierarchies. Flat custom lists are handled in the extension layer
  • Minimum 2 levels per dimension, maximum 15
  • When modifying hierarchy levels, add or remove from the middle — never from the leaf
  • Click Apply Changes after every modification. If you need to start over, use Restore default

Step 1 — Rename every model with the standardized naming convention

Give every model a unique name before you touch any dimension. Each name uses today's date so this run is unique in the shared training tenant.

  1. From the Workspace, select View All Models.
  2. Select Manage models.
  3. Select IFP v2.0.0 TRAIN - Base - Administration, click Rename, enter admin_Exercise 2_YYYYMMDD in the Model Name field, and click OK.
  4. Back at the Model Management screen, select the model named IFP v2.0.0 TRAIN - Base - CapEx, click Rename, enter capex_Exercise 2_YYYYMMDD in the Model Name field, and click OK.
  5. Back at the Model Management screen, select the model named IFP v2.0.0 TRAIN - Base - Headcount, click Rename, enter headcount_Exercise 2_YYYYMMDD in the Model Name field, and click OK.
  6. Back at the Model Management screen, select the model named IFP v2.0.0 TRAIN - Base - Financial Planning, click Rename, enter fp_Exercise 2_YYYYMMDD in the Model Name field, and click OK.
  7. Back at the Model Management screen, select the model named Application Training, click Rename, enter ifp-app_Exercise 2_YYYYMMDD in the Model Name field, and click OK.
  8. Click Close to exit the Model Management dialog.

Quick check

You finished Exercise 1 yesterday and the tenant still has models named admin_Exercise 1_20260613. Why does that not conflict with today's Exercise 2 generation?

Correct. The whole point of [ModelType]_Exercise ##_YYYYMMDD is that the exercise number and the date together guarantee a unique name even when you re-run an exercise. Yesterday's Exercise 1 models stay in the tenant without colliding with today's Exercise 2 models.
Not quite. The right answer is C. The naming pattern includes the exercise number and the date precisely so prior exercises don't collide with the current one. Anaplan does not auto-suffix duplicates — generation just fails.

Step 2 — Configure the eight standard dimensions

This scenario has a deeply hierarchical Product dimension and uses Vendor as a Projects placeholder. Click Apply Changes after each dimension before moving on.

  1. In the Configurator, open the Dimensions section.
  2. For Entity, keep it required, leave the name as-is, and set # of levels to 2. Click Apply Changes.
  3. For Department, keep it required, leave the name as-is, and set # of levels to 2. Click Apply Changes.
  4. For Product, keep it required, leave the name as Product, and set # of levels to 5 for the multi-product portfolio. Click Apply Changes.
  5. For Job, keep it required, leave the name as-is, and set # of levels to 3. Click Apply Changes.
  6. For Vendor, keep it required, rename it to Projects, and set # of levels to 2. Click Apply Changes.
  7. For Functional Area, keep it required, leave the name as-is, and set # of levels to 3. Click Apply Changes.
  8. For Customer, mark it not required. Click Apply Changes.
  9. For Location, mark it not required. Click Apply Changes.
End-of-step screenshot to capture Configurator Dimensions section after Step 2 — Entity (2), Department (2), Product (5), Job (3), Projects (2, renamed from Vendor), Functional Area (3) all marked required; Customer and Location marked not required.

Once all eight dimensions are configured, run the check below.

Dimensions look good.

  • Entity — required, 2 levels
  • Department — required, 2 levels
  • Product — required, 5 levels
  • Job — required, 3 levels
  • Projects (renamed from Vendor) — required, 2 levels
  • Functional Area — required, 3 levels
  • Customer, Location — not required

Dimension scope matches the 3-statement insurance scenario. You're cleared to move on to Step 3.

Quick check

Why does this scenario set the Product dimension to 5 levels rather than the default?

Correct. Level count is dictated by the client's actual reporting structure — not by a rule of the Configurator. Here the insurance scenario calls for detailed product profitability, so five levels of Product hierarchy is the right fit.
Not quite. The right answer is B. There's no hard rule tying level count to anything else in the configuration — you set the depth to match what the client actually needs to plan and report on.

Step 3 — Add the CapEx List custom dimension

The insurance scenario tracks capital projects through a flat custom list. Flat lists are added in the Custom Lists section, not as standard dimensions.

  1. Open the Custom Lists section in the Configurator.
  2. Add a new custom list named CapEx List with type Flat — this is the placeholder for capital project tracking. Click Apply Changes.
End-of-step screenshot to capture Custom Lists section with the CapEx List entry added as a Flat list.

Once the CapEx List is added, run the check below.

Custom list looks good.

  • CapEx List — added as a Flat custom list

Capital project tracking placeholder is in place. You're cleared to move on to Step 4.

Quick check

Why isn't CapEx List configured as a standard dimension with hierarchy levels?

Correct. The Configurator's standard dimensions are always composite (multi-level). Anything you want as a flat list goes into Custom Lists, and the extension exercises wire those lists into the model as line-item dimensions afterwards.
Not quite. The right answer is A. The shape — composite vs. flat — decides where a list lives in the Configurator. Composite goes in Dimensions; flat goes in Custom Lists.

Step 4 — Answer the top-level configuration questions

The top-level section orients the Configurator before drilling into each model.

  1. Open the Top-Level Questions section in the Configurator.
  2. Set Use the Geography dimension? to No — not required by this scenario.
  3. Set Use the Functional Area dimension? to Yes — the scenario keeps Functional Area required in Step 2.
  4. Click Apply Changes to save both answers.
End-of-step screenshot to capture Top-Level Questions section with Use the Geography dimension set to No and Use the Functional Area dimension set to Yes.

Once both top-level answers are saved, run the check below.

Top-level answers look good.

  • Use the Geography dimension → No
  • Use the Functional Area dimension → Yes

Orientation questions are aligned with the dimensions you configured in Step 2. You're cleared to move on to Step 5.

Quick check

A teammate sets Use the Functional Area dimension? to No but kept Functional Area marked required in Step 2. What happens?

Correct. The Configurator doesn't reconcile contradictions across sections for you. A dimension marked required but turned off at the top-level is the same shape of inconsistency you'll see surfaced as a LINE_ITEM_FORMULA error in Error Log Review.
Not quite. The right answer is C. The Configurator captures each answer literally — it's on you to keep the dimension list and the top-level answers in agreement.

Step 5 — Answer the Financial Planning questions

Three-statement scope is the headline of this exercise. Balance Sheet and Cashflow are both turned on, and the expense planning dimensionality matches the dimensions you configured in Step 2.

  1. Open the Financial Planning section in the Configurator.
  2. Set Top-down targets? to No.
  3. Set Balance Sheet? to Yes — required for 3-statement planning.
  4. Set Cashflow reporting? to Yes — required for 3-statement planning.
  5. Set Dimensionality for expense planning to Department, Vendor.
  6. Set Dimensionality for itemized expense planning to Department.
  7. Set Expense allocations? to No.
  8. Click Apply Changes to save all six answers.
End-of-step screenshot to capture Financial Planning section with Top-down targets No, Balance Sheet Yes, Cashflow reporting Yes, Dimensionality for expense planning Department, Vendor, Itemized dimensionality Department, Expense allocations No.

Once all Financial Planning answers are saved, run the check below.

Financial Planning answers look good.

  • Top-down targets → No
  • Balance Sheet → Yes
  • Cashflow reporting → Yes
  • Dimensionality for expense planning → Department, Vendor
  • Dimensionality for itemized expense planning → Department
  • Expense allocations → No

Financial Planning is set for full 3-statement scope. Configuration for Exercise 2 is complete.

Quick check

What changes about the generated model when you set Balance Sheet and Cashflow reporting both to Yes?

Correct. These two answers expand the Financial Planning model's scope from a P&L-only model (what Exercise 1 produced with Expense Only margin planning) to a full 3-statement structure. The generated model now carries balance-sheet line items and a cashflow rollup.
Not quite. The right answer is A. Balance Sheet and Cashflow widen the Financial Planning model's scope to full 3-statement planning — that's the headline change versus Exercise 1.

You're done with the configuration

Exercise 2 is configured for full 3-statement planning with deep product hierarchy and a CapEx placeholder. Leave the Configurator alone for now and head to Exercise 3, which narrows scope back to OpEx + headcount but adds dimension attributes and a richer custom-list set.
