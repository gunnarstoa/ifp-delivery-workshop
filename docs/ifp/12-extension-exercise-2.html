Extension Exercise 2: Channel Revenue Planning

Add a Channel dimension, a channel-level revenue module, and conditionally integrate it into Gross Profit Planning

Extensions⏱ 60 minGuided practice

What you'll do

  • Create the EXT Channel flat list and a new channel-level revenue planning module
  • Register Channel Planning as a new planning method and turn it on for a target product
  • Add three line items to INP Gross Profit Planning and write a conditional aggregation that only fires when the planning method is selected
  • Update the LISS so the new line items appear in the UX, and update existing aggregation formulas so channel revenue rolls into Final Amount and SUM: Revenue
  • Verify the full flow end-to-end for the target product in the UX

The scenario

The client wants channel-level revenue planning — a new dimension on top of the existing revenue model so they can plan Price × Volume per channel and roll the result back into Gross Profit Planning. The catch: not every product should be planned this way. Channel Planning needs to be an opt-in choice per product, and the aggregation back into Gross Profit Planning has to respect that opt-in.

Before you start: how extensions stay safe

⚠ Rules to follow throughout the exercise
  • EXT prefix on everything you create. Lists and modules added as extensions get the EXT prefix so they're easy to tell apart from generated content during upgrades.
  • Register new planning methods. Every new way of planning has to land in the Planning Methods list — otherwise the UX won't recognize it as a valid option.
  • Conditional formulas. When the new method isn't selected, the conditional should return 0 or blank — never a stale aggregation. IF(Use Channel Planning Method?, …, 0) is the safe pattern.
  • Update the LISS, or the UX won't see it. A line item that isn't checked in LISS: Margin Filter stays invisible on the relevant UX page.
  • Documentation is the upgrade plan. Every line-item and formula change goes in the module Notes and the offline change log. When Anaplan ships a new IFP base, those notes are how you re-apply the extension.

Step 1 — Create the EXT Channel list

The new dimension is a flat list — just discrete channel names with one top-level aggregator.

  1. In the Data Model editor, create a new list named EXT Channel. Use Flat structure (no parent levels).
  2. Add the top-level item All Channels — this aggregates every individual channel.
  3. Add three sample channel items: Channel 1, Channel 2, Channel 3.
  4. Save the list.

Step 2 — Create the channel revenue planning module

The new module carries Price, Volume, and a calculated Total. It's dimensioned across the existing Gross Profit Planning dimensions plus the new EXT Channel.

  1. Create a new module named EXT INP Channel Revenue Planning.
  2. Set the module's dimensions: Time, every dimension from INP Gross Profit Planning (Product, Job, Entity, etc.), plus the new EXT Channel list.
  3. Add three line items:
    • Price — unit price per channel
    • Volume — units sold per channel
    • Total — calculated revenue
  4. On the Total line item, set the formula Price * Volume.
  5. Save the module.
Quick check

Why does the new list use the prefix EXT (EXT Channel) instead of just Channel?

Correct. Convention, not technical requirement. When a new IFP version regenerates the model, the generated content reappears at its original names and your extensions don't. The EXT prefix is the visible marker that says "this came from an extension — re-apply it from your change log."
Not quite. The right answer is B. The prefix has no technical effect — it's the naming convention that makes generated vs. extension content distinguishable, especially when upgrades regenerate the model and you need to know what to re-apply.

Step 3 — Register Channel Planning as a planning method

Every new way of planning has to be registered in the Planning Methods list, and then made available to the specific products that should be planned that way.

  1. Open the module SYS by Planning Methods.
  2. In the List Planning Methods, add a new item: Channel Planning.
  3. Save.
  4. Open the module ADMIN Define Product to Planning Method.
  5. For your target product — for example, Service Offering 1 — add Channel Planning to its list of available planning methods.
  6. Save.

Step 4 — Add channel planning line items to Gross Profit Planning

The new line items live in INP Gross Profit Planning and turn the channel-level total into a number Gross Profit Planning can use.

  1. Open the module INP Gross Profit Planning in blueprint view.
  2. Add three new line items before the existing Final Amount line item:
    • Channel Planning — Heading 1, no data (organizational marker only)
    • Use Channel Planning Method? — Boolean (planner checks Yes to opt in for this product). Copy settings from a similar Boolean such as Use COGS Pct of Revenue Method?.
    • Channel Planning Revenue — same format as Final Amount; this is the aggregated channel revenue that rolls back into the parent module.
  3. On the Channel Planning Revenue line item, write a conditional formula along these lines (adjust for your model's exact naming):
    • IF(Use Channel Planning Method?, SUM(EXT INP Channel Revenue Planning.Total), 0)
  4. Document the formula in the module's Notes field and in your offline change log.

Step 5 — Update the LISS and the existing aggregation formulas

The new line items need to be registered in the LISS so the UX shows them, and the existing aggregation formulas need to pick up the channel-revenue contribution.

  1. Open LISS: Margin Filter.
  2. Check the Boolean for each of the three new line items: Channel Planning (heading), Use Channel Planning Method?, and Channel Planning Revenue.
  3. Save.
  4. In INP Gross Profit Planning, open the formula for Final Amount. Add + Channel Planning Revenue to its existing components. (Because Channel Planning Revenue is already conditional, you don't need to wrap it again here.)
  5. If a SUM: Revenue line item exists, add + Channel Planning Revenue to its formula too so channel-based revenue shows up in revenue summaries.
  6. Document both updated formulas — old vs. new — in the module Notes and your offline change log.
Quick check

Why is Channel Planning Revenue wrapped in IF(Use Channel Planning Method?, …, 0) instead of just summing the EXT module directly?

Correct. The IF is what makes the extension safe to roll into Final Amount. Products that aren't planned by channel return 0; products that opt in return the SUM across channels. Without the IF, every product would silently pick up a channel-revenue contribution — exactly the kind of silent double-count that's hard to find later.
Not quite. The right answer is C. The conditional respects the opt-in flag on each product. Anaplan doesn't require the IF technically — but the business logic does, otherwise non-channel products would double-count revenue.

Step 6 — Verify the full flow in the UX

The extension isn't finished until a planner can walk through it end-to-end for a real product. Don't skip this — it's the only check that proves the formula, the LISS, and the planning-method registration all line up.

  1. Open the UX app and navigate to the Revenue & COGS - planning page for Service Offering 1.
  2. Confirm the new Use Channel Planning Method? Boolean appears and is editable.
  3. Set Use Channel Planning Method? to Yes. The interface should adjust to reveal channel planning.
  4. Navigate to the Channel Revenue Planning page. Enter sample Price and Volume for each of the three channels.
  5. Return to Revenue & COGS - planning. Confirm Channel Planning Revenue shows the SUM across channels and that Final Amount picks it up.
  6. Spot-check a second product where Use Channel Planning Method? is still No. Confirm its Channel Planning Revenue is 0 and Final Amount does not include any channel contribution.
  7. If something's missing: verify the line items are in LISS: Margin Filter, that the formula syntax matches your model's exact line-item names, and that the EXT Channel list is referenced correctly in the EXT module's dimensions.

Once the opt-in product shows channel revenue rolling into Final Amount and the non-opt-in product shows 0, run the check below.

Channel Revenue Planning extension wired up end-to-end.

  • EXT Channel list → flat list with All Channels + 3 channel items
  • EXT INP Channel Revenue Planning → module with Price, Volume, Total = Price × Volume
  • Channel Planning → registered as a planning method and enabled on the target product
  • INP Gross Profit Planning → 3 new line items; Channel Planning Revenue uses the conditional IF formula
  • LISS: Margin Filter → all 3 new line items checked in
  • Aggregations → Final Amount (and SUM: Revenue if present) include Channel Planning Revenue
  • UX verification → opt-in product rolls channel revenue into Final Amount; non-opt-in product returns 0
  • Documentation → module Notes + offline change log updated

Extension is complete and verified for both opt-in and non-opt-in products. You've finished the extension exercises.

⚠ Upgrade consideration

When Anaplan ships a new IFP version, every extension you've built has to be re-applied to the newly generated model. Your offline change log — original and updated formula syntax, line-item names, LISS registrations, planning-method registrations — is the script for that re-application. Without it, every upgrade becomes detective work.

Key takeaways

  • Extensions add, they don't replace — generated structure stays as it was; you append new line items and adjust formulas around them.
  • Planning Methods organize user choices — every new planning approach has to be registered, otherwise the UX has no way to expose it.
  • Line-Item Subsets control visibility — new content only appears on the UX page when it's checked into the LISS.
  • Formulas tied to opt-in flags must be conditional — otherwise non-opted-in records silently pick up extension data.
  • Documentation is the upgrade plan — every change goes in module Notes and an offline change log.
  • Test end-to-end — data entry, conditional formula, aggregation, summary all in the UX, with both opt-in and non-opt-in products spot-checked.
