IFP Delivery Partner Workshop

Exercise 1: OpEx and Headcount Planning

Configure your first IFP application for a manufacturing company

Configuration⏱ About 30 minsGuided practice

What you'll do

  • Apply the standardized naming convention so every generated model has a unique name in your tenant
  • Configure the eight standard dimensions for an OpEx + headcount scope
  • Answer the top-level, Financial Planning, and Headcount configuration questions
  • Internalize the six rules that keep your configuration generation-ready

The scenario

A manufacturing company needs to plan operating expenses and headcount across multiple departments and cost centers. No revenue, COGS, balance sheet, or capital expenditure are required in this configuration. You'll set up just enough to support OpEx and headcount planning — the rest of the surface area (CapEx, Revenue, Balance Sheet) comes in later exercises.

Before you start: the six rules

These rules make the difference between a configuration that generates cleanly and one that fails halfway through. Read them once now, then refer back as you work.

⚠ Rules to follow throughout the exercise
  • Use a standardized naming convention for every model and app — duplicate names in the tenant cause generation to fail
  • Renames must be no longer than the original dimension name, especially "Department"
  • All standard dimensions are composite hierarchies. Flat custom lists are handled in the extension layer
  • Minimum 2 levels per dimension, maximum 15
  • When modifying hierarchy levels, add or remove from the middle — never from the leaf
  • Click Apply Changes after every modification. If you need to start over, use Restore default

Step 1 — Rename every model with the standardized naming convention

Before you touch any dimension, give every model a unique name. The names below use today's date so each run is unique in the shared training tenant — type each one exactly as shown.

  1. From the Workspace, select View All Models.
  2. Select Manage models .
  3. Select IFP v2.0.0 TRAIN - Base - Administration, click Rename, enter admin_Exercise 1_YYYYMMDD in the Model Name field, and click OK.
  4. Back at the Model Management screen, select the model named IFP v2.0.0 TRAIN - Base - CapEx, click Rename, enter capex_Exercise 1_YYYYMMDD in the Model Name field, and click OK.
  5. Back at the Model Management screen, select the model named IFP v2.0.0 TRAIN - Base - Headcount, click Rename, enter headcount_Exercise 1_YYYYMMDD in the Model Name field, and click OK.
  6. Back at the Model Management screen, select the model named IFP v2.0.0 TRAIN - Base - Financial Planning, click Rename, enter fp_Exercise 1_YYYYMMDD in the Model Name field, and click OK.
  7. Back at the Model Management screen, select the model named Application Training, click Rename, enter ifp-app_Exercise 1_YYYYMMDD in the Model Name field, and click OK.
  8. Click Close to exit the Model Management dialog.

Correct. Names must be unique within the tenant — generation halts halfway through if it tries to create an artifact whose name already exists. The pattern [ModelType]_Exercise ##_YYYYMMDD keeps every run unique even when the same exercise is repeated.

Step 2 — Configure the eight standard dimensions

IFP ships with eight standard dimensions. Mark the ones this scenario needs, rename four of them, and set the right number of hierarchy levels for each. Click Apply Changes after each dimension before moving to the next.

  1. In the Configurator, open the Dimensions section.
  2. For Entity, keep it required, leave the name as-is, and set # of levels to 2. Click Apply Changes.
  3. For Department, keep it required, rename it to CostCenter, and set # of levels to 3. Click Apply Changes.
  4. For Product, mark it not required. Click Apply Changes.
  5. For Job, keep it required, rename it to Roles, and set # of levels to 3. Click Apply Changes.
  6. For Vendor, keep it required, rename it to Suppliers, and set # of levels to 2. Click Apply Changes.
  7. For Functional Area, keep it required, rename it to HR Org, and set # of levels to 3. Click Apply Changes.
  8. For Customer, mark it not required. Click Apply Changes.
  9. For Location, mark it not required. Click Apply Changes.

This scenario also needs two custom lists handled in the extension layer (not in the Configurator).

  1. Open the Custom Lists section. Add Projects as a flat list — you'll use it for project tracking in the extension exercise.
  2. Add Country as a flat list — it's a supporting attribute list for later use.
Screenshot to capture Configurator Dimensions section after Step 2 is complete — Entity (2 levels), CostCenter (3), Roles (3), Suppliers (2), HR Org (3) all marked required; Product, Customer, Location all marked not required.

Once all eight dimensions are configured and both custom lists are added, run the check below.

Dimensions look good.

  • Entity — required, 2 levels
  • CostCenter (renamed from Department) — required, 3 levels
  • Roles (renamed from Job) — required, 3 levels
  • Suppliers (renamed from Vendor) — required, 2 levels
  • HR Org (renamed from Functional Area) — required, 3 levels
  • Product, Customer, Location — not required
  • Projects and Country added as flat custom lists

Dimension scope matches the scenario. You're cleared to move on to Step 3.

Quick check

Your client needs to track spend by project code. How do you handle "Project" in this exercise?

Correct. All standard dimensions in the Configurator are composite hierarchies. Anything that needs to be a flat list — like Projects or Country here — is added in the extension layer instead. You'll see this in action in the Extensions exercises.
Not quite. The right answer is C. Standard dimensions are always composite hierarchies. Flat lists like Projects belong in the extension layer, not in the Configurator.

Step 3 — Answer the top-level configuration questions

The Configurator asks a handful of orientation questions before drilling into each model.

  1. Open the Top-Level Questions section in the Configurator.
  2. Set Multi-currency support? to Yes — multi-entity reporting requires it.
  3. Set Use the Location dimension? to No — not required by this scenario.
  4. Set Use the Functional Area dimension? to Yes — repurposed as "HR Org" (you renamed it in Step 2).
  5. Click Apply Changes to save all three answers.
End-of-step screenshot to capture Top-Level Questions section with Multi-currency support set to Yes, Use the Location dimension set to No, and Use the Functional Area dimension set to Yes.

Once the three top-level answers are saved, run the check below.

Top-level answers look good.

  • Multi-currency support → Yes
  • Use the Location dimension → No
  • Use the Functional Area dimension → Yes

All three orientation questions are answered for this scenario. You're cleared to move on to Step 4.

Quick check

Why is Multi-currency support set to Yes at the Top-Level question for this scenario?

Correct. A manufacturing company planning across multiple departments typically reports across multiple entities — that's the trigger for multi-currency. If everything were in one entity in one currency, you could safely leave it off.
Not quite. The right answer is B. Multi-currency support is needed for multi-entity reporting — when a single model serves entities operating in different currencies.

Step 4 — Answer the Financial Planning questions

The Financial Planning model is the OpEx engine for this scenario. Pay attention to Margin Planning → Expense Only — that's what scopes the model to OpEx and skips revenue.

  1. Open the Financial Planning section in the Configurator.
  2. Set Multi-currency support? to Yes.
  3. Set Top-down targets? to No.
  4. Set Margin Planning? to Expense Only — this scopes the model to OpEx and skips revenue, COGS, and balance-sheet planning.
  5. Set Expense planning? to Yes.
  6. Set Dimensionality for expense planning to Department, Vendor.
  7. Set Dimensionality for itemized expense planning to Department.
  8. Set Expense allocations? to No.
  9. Click Apply Changes to save all seven answers.
End-of-step screenshot to capture Financial Planning section with all seven answers set per the scenario: Multi-currency Yes, Top-down targets No, Margin Planning Expense Only, Expense planning Yes, Dimensionality Department, Vendor, Itemized dimensionality Department, Expense allocations No.

Once all seven Financial Planning answers are saved, run the check below.

Financial Planning answers look good.

  • Multi-currency support → Yes
  • Top-down targets → No
  • Margin Planning → Expense Only
  • Expense planning → Yes
  • Dimensionality for expense planning → Department, Vendor
  • Dimensionality for itemized expense planning → Department
  • Expense allocations → No

All seven Financial Planning answers match the OpEx-only scope. You're cleared to move on to Step 5.

Quick check

Why did you set Margin Planning to Expense Only in this exercise?

Correct. Margin Planning controls the scope of the Financial Planning model. Expense Only is right when the client only needs OpEx — it strips out the revenue and balance-sheet pieces so generation only builds what you'll actually use. Exercise 2 will widen the scope.
Not quite. The right answer is A. Margin Planning controls the model's scope — Expense Only matches a client who needs OpEx but not revenue, COGS, or balance-sheet planning. There's no universal default; the scenario dictates the answer.

Step 5 — Answer the Headcount questions

The Headcount model handles workforce planning.

  1. Open the Headcount section in the Configurator.
  2. Set Multi-currency support? to Yes.
  3. Set Term for Job? to Roles — this must match the rename you applied to the Job dimension in Step 2.
  4. Click Apply Changes to save both answers.
End-of-step screenshot to capture Headcount section with Multi-currency support set to Yes and Term for Job set to Roles.

Once both Headcount answers are saved, run the check below.

Headcount answers look good.

  • Multi-currency support → Yes
  • Term for Job → Roles

Both Headcount answers are set, and the term matches the Job dimension's rename in Step 2.

Quick check

Why must the Term for Job in the Headcount section match the rename you applied to the Job dimension in Step 2?

Correct. The Headcount model's formulas and import actions reference whichever term you chose for Job. If the rename and the term don't match, you'll see LINE_ITEM_FORMULA errors after generation — exactly the kind covered in Error Log Review.
Not quite. The right answer is B. The Headcount model uses "Job" as the source of truth for references inside its formulas. If you rename the dimension in Step 2 but leave the Headcount term as the default, generation produces broken references — the kind that show up in the error log review later.

You're done with the configuration

You've now set up a complete OpEx + headcount configuration. You haven't generated the model yet — that comes in Model Generation. For now, leave the Configurator alone and head to Exercise 2, which widens the scenario to include revenue and CapEx.
