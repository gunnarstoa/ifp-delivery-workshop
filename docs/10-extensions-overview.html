What Extensions Are

Extensions are customer-specific additions or modifications to the IFP base model that the configurator cannot produce. They fall into three categories — and the category determines the risk and the approach.

Extension TypeWhat It IsUpgrade RiskWhen to Use
ConfigureUse the configurator to enable a built-in capabilityNone — this is supportedAlways try this first. If the configurator has a setting for it, use the configurator.
Extend / ConnectAdd new line items, modules, or lists without touching base model formulasLow — new items are not overwritten by upgradesWhen the customer needs something the configurator doesn't cover, but you can add it without changing base model logic.
Amend / Modify / DeleteChange an existing base model formula, list, or moduleHigh — changes may be lost or broken on upgradeLast resort. Document every override with a data tag. Maintain external documentation.

7 Best Practices for IFP Extensions

1

Configure First

Before touching the model, check the configurator. IFP 2.0 has 44 configuration questions. The answer to a customer requirement may already be a configurator setting.

2

Extend or Connect Next

If configuration doesn't cover it, add to the model without modifying base model components. New modules, new line items, new lists — these are safe to add.

3

Modify With Care

When you must change base model formulas or structures, treat it as the highest-risk action. Document it. Tag it. Test it. And plan for what happens at the next IFP upgrade.

4

Document Everything

Create two data tags in every IFP model before your first extension: Extension and Formula Override. Tag every line item you add or modify. This is how you track what needs re-validation after upgrades.

5

Test Before Finalizing

Every extension must be tested end-to-end: model formula → UX display → downstream aggregation. Do not sign off on an extension until you have verified it produces the correct output in the UX.

6

Preserve the Upgrade Path

Tag formula overrides so you know exactly what to re-check when Anaplan releases an IFP update. When an upgrade arrives, run your formula override tag filter — that's your review list.

7

External Documentation Is Critical

In-model notes and data tags are not sufficient. Maintain an external design spec that lists every extension, what it does, which line items it touches, and what the upgrade impact is.

Data Tags — Create These First

⚠ Tag Before You Build

Before making any extension changes to an IFP model, create two data tags: Extension (for all new line items you add) and Formula Override (for any base model formula you change). These tags are the only in-model record of what you've changed. Without them, a future consultant has no way to know.

⚠ Formula Overrides and Upgrades

Formula overrides on base model line items are the highest-risk extension action. When Anaplan releases an IFP update, formula overrides may be lost or broken. Always tag them with the Formula Override data tag and document them externally. After any IFP upgrade, filter by the Formula Override tag and re-validate each one.
