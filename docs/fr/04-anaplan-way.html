La méthodologie de livraison IFP en six phases

La livraison IFP suit les phases standard de livraison d'applications d'Anaplan, adaptées au workflow de génération piloté par l'Application Framework Configurator. Comprendre les phases vous aide à fixer les bonnes attentes auprès des clients et à savoir quelles activités relèvent de quelle phase.

📚 Document de référence

The Anaplan Way for Applications Methodology — Méthodologie complète pour livrer des implémentations applicatives structurées.

1

Pre-Project

La phase de préparation de la mission. Les livrables comprennent :

  • Session Zero — découverte de la configuration avec le client
  • Feuille de travail de configuration complétée (les 44 questions répondues)
  • Nommage des Hiérarchies et nombre de niveaux déterminés
  • Disponibilité des données confirmée : source des actuals, flux Headcount, configuration des devises
  • Workspace provisionné, accès Workspace Admin confirmé pour l'équipe de livraison

2

Config

La phase de l'Application Framework Configurator. Le consultant livraison :

  • Saisit toutes les réponses de configuration dans l'Application Framework Configurator
  • Passe en revue les questions top-level ; financial planning ; headcount ; configurations capex
  • Lance la Génération du modèle
  • Examine les Error Logs

3

Test & Deploy

Phase de validation après une Génération propre :

  • Peupler les Listes/Hiérarchies et charger des données d'actuals d'exemple via les pipelines ADO
  • Vérifier ponctuellement les Line Items calculés dans les modules CALC
  • Valider les méthodes de planification dans les pages UX
  • UAT avec les utilisateurs clés de l'équipe client

4

Extension Foundation

Concevoir et cadrer les extensions spécifiques au client :

  • Identifier les écarts entre le modèle de base IFP et les exigences client
  • Classifier chaque écart : Configurer vs. Étendre vs. Modifier
  • Dimensionner chaque extension (S/M/L/XL)
  • Créer les tags d'extension dans le modèle (Extension + Formula Override)

5

Iterative Extension

Construire les extensions par ordre de priorité :

  • Chaque extension : concevoir → construire → tester → documenter
  • Taguer chaque Line Item que vous ajoutez ou modifiez
  • Documenter dans une spécification de conception externe (et pas seulement des notes dans le modèle)
  • Tests de régression sur les fonctionnalités du modèle de base après chaque extension

6

Extension Test & Deploy

Validation finale et transfert :

  • UAT complète avec toutes les extensions en place
  • Test de performance avec des volumes de données de production
  • Documentation de transfert : journal des extensions, mapping des formula overrides, checklist de mise à niveau
  • Formation des administrateurs client

ℹ Notes spécifiques aux phases IFP

Session Zero est critique. Contrairement aux constructions de modèles sur mesure, les décisions de configuration IFP prises lors de Session Zero sont difficiles voire impossibles à changer après la Génération. Les niveaux de Hiérarchies, la structure de géographie et la configuration des entités doivent être figés avant que quiconque ne touche à l'Application Framework Configurator. Une Dimension oubliée lors de Session Zero = une régénération complète pendant Test & Deploy.

Deux types de clients

🌱 Greenfield

  • Nouveau Workspace, aucun modèle Anaplan existant
  • IFP est la première application
  • Aucune complexité d'intégration avec des modèles existants
  • Mise en place complète d'ADO à partir de zéro
  • Risque : la découverte en Session Zero est la seule contrainte — faites-la correctement

🔗 Workspace existant

  • IFP déployé aux côtés de modèles en production existants
  • Le Workspace Admin doit gérer les conflits de Listes
  • ADO peut devoir cohabiter avec des actions d'import existantes
  • Le nommage des Listes ne doit pas entrer en conflit avec les Listes existantes
  • Risque : la Génération peut affecter les paramètres du Workspace existant
