Ce qu'est l'Application Framework Configurator

L'Application Framework Configurator est l'interface unique pour toutes les décisions de configuration IFP. Il présente 44 questions de configuration réparties en trois catégories : questions top-level (23), configurations financial planning (17) et configurations headcount (3), plus des paramètres supplémentaires. Les réponses à ces questions déterminent ce qui est généré — structure des Modules, Hiérarchies de Listes, logique des formules et méthodes de planification disponibles.

L'Application Framework Configurator est accessible via une URL spécifique : remplacez « home » dans l'URL Anaplan de votre navigateur par « applications » (pas via la navigation normale du Workspace, qui n'affiche pas encore le lien Applications en v2.1.0). Par exemple, si votre URL d'accueil Anaplan est « https://us1a.app.anaplan.com/home », changez-la en « https://us1a.app.anaplan.com/applications ». À mesure que l'application atteindra la disponibilité générale, ce lien apparaîtra dans la navigation standard.

i Bonne pratique : Générez d'abord une copie de base

Avant de faire des changements de configuration, générez d'abord l'application avec tous les paramètres par défaut. Cela vous donne une copie de base — un modèle dans lequel toutes les formules ont été générées correctement, avec les noms de Hiérarchies par défaut et les Dimensions par défaut intactes. Si vous obtenez plus tard une erreur de formule dans votre Génération configurée, vous pouvez retrouver la bonne formule dans la copie de base et la recopier. Étiquetez clairement cette Génération (par ex. « Base - sans changement »).

⚠ La configuration est figée à la Génération

Une fois la Génération lancée, de nombreuses décisions structurelles ne peuvent plus être modifiées sans régénérer le modèle. Les niveaux de Hiérarchies, la structure des entités et la configuration de géographie doivent être figés avant la Génération. Faites-les correctement dès Session Zero — ce sont les choses les plus difficiles à changer ensuite.

Aperçu de l'Application Framework Configurator (v2.1)

Sections de configuration

Hierarchies

La section Hierarchies définit la structure dimensionnelle du modèle généré. Les décisions prises ici se répercutent sur chaque Module et ne peuvent pas être modifiées après la Génération sans une régénération complète.

  • Location : combien de niveaux ? (par ex. Global → Region → Country → City). Quels niveaux sont des niveaux de planification vs. uniquement d'agrégation ?
  • Functional Area : quelles fonctions sont dans le périmètre ? (Gross Profit, OpEx, Headcount, Balance Sheet). Chaque domaine activé génère des Modules supplémentaires.
  • Entity : combien de niveaux d'entités ? Quelle est l'entité de consolidation de haut niveau ?
  • Product : la planification au niveau produit est-elle requise ? Niveaux de Hiérarchie produit ?

Paramètres de dimensionnalité

ℹ Impact des questions de dimensionnalité sur « Applies To »

Ces questions (13-17) impactent le « Applies To » lors de la génération des Modules. Par défaut, toutes les Dimensions optionnelles sont sélectionnées car on a supposé que toutes les Dimensions seraient utilisées. Si certaines Dimensions ne sont pas nécessaires, décochez-les.

Décocher des Dimensions réduit la complexité des Modules générés et améliore les performances en excluant des combinaisons de Dimensions inutiles des calculs.

Nuances clés du Configurator (apprises sur le terrain)

  • Le nom de Hiérarchie et la question de l'Application Framework Configurator doivent correspondre : en v2.1.0, le nom de la Hiérarchie dans la section Hierarchies et le renommage dans les Configuration Questions ne sont PAS liés. Vous devez mettre à jour les deux. Une correction est attendue dans une future version.
  • Pas d'espaces en fin de nom : les espaces en fin de noms de Hiérarchies renommées ont causé des erreurs de Génération. Vérifiez que les noms n'ont pas d'espaces avant ou après.
  • Le niveau feuille doit rester en bas : lorsque vous supprimez un niveau, retirez le niveau intermédiaire et gardez la feuille en bas. Lorsque vous ajoutez un niveau, le système l'ajoute en bas — glissez-le au-dessus de la feuille d'origine. La feuille d'origine ne peut pas être renommée avec le même nom que le nouveau niveau.
  • Hierarchies et Configuration Questions sont séparées : il y a deux questions par Dimension renommée — l'une crée la Liste, l'autre crée les Modules. La seconde est en minuscules. Vous devez mettre à jour les deux.
  • Restore Default par Hiérarchie : chaque Hiérarchie a son propre bouton « Restore Default ».
  • Ordre Configurator vs Hierarchies : vous pouvez compléter les Hiérarchies en premier ou les questions de configuration en premier — aucun ordre n'est requis.
  • Générez CapEx même si non nécessaire maintenant : les Modules Polaris n'utilisent pas beaucoup de mémoire tant qu'ils ne contiennent pas de données. Générez CapEx maintenant pour éviter une future régénération si le client l'ajoute plus tard.
  • Les Dimensions personnalisées (Listes plates) sont des extensions, pas des Hiérarchies : les Dimensions de Listes plates (par ex. Channel) ne se configurent pas dans la section Hierarchies — elles se construisent en tant qu'extensions post-Génération.

Processus de Génération

Après avoir complété toutes les questions de configuration et les paramètres de Hiérarchies :

  1. Cliquez sur Generate depuis l'aperçu de l'Application Framework
  2. Saisissez une description pour cette Génération (par ex. « Base — sans changement » ou « Nom Client — 3 niveaux d'entité, sans géographie »)
  3. Sélectionnez le Workspace cible
  4. La Génération s'exécute dans le cloud — environ 20 à 40 minutes. Suivez la progression sur la page principale /applications plutôt qu'à l'intérieur de l'Application Framework (l'indicateur de statut à l'intérieur du framework peut être en retard)
  5. Une fois terminé, tous les éléments s'affichent en vert. Les éléments en rouge sont des erreurs. Les éléments en orange sont des avertissements.
  6. Examinez l'Error Log. Voir la section « Revue de l'Error Log » pour plus de conseils.

Avant l'exercice

i Préparation

Avant de configurer l'Application Framework pour votre client, identifiez les questions de portée clés : niveaux d'entité, structure de géographie, fonctions dans le périmètre et configurations spéciales. Travaillez avec le client pour clarifier les décisions ambiguës avant de commencer.
