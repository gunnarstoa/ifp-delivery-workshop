Les Error Logs — ce que c'est

Les Error Logs sont générés par l'Application Framework après la Génération du modèle. Ils listent les problèmes de formules et de configuration qui surviennent lorsque le modèle généré diffère du modèle de base. L'Error Log est accessible en bas à droite de la page d'aperçu de l'Application Framework et peut être exporté vers Excel.

Les Error Logs sont un nouveau contenu introduit dans IFP 2.0 — les versions IFP antérieures ne disposaient pas de ce reporting d'erreurs systématique. Apprendre à les lire efficacement est une compétence clé de praticien.

Référence des types d'erreur

Type d'erreurSignificationAction
LINE_ITEM_FORMULAUne formule référence un Line Item qui n'existe pas ou a été renommé dans le modèle généré — souvent dû au renommage de Hiérarchies ou à des choix de configurationTrouver le Line Item par code (préfixe avec underscore), comparer la formule au modèle de base, corriger ou recopier-ajuster depuis la base
LIST_PROPERTY_FORMULALa formule référence une propriété de Liste qui n'existe pas ou a été renomméeTrouver le Line Item par code, comparer la formule au modèle de base, mettre à jour la référence à la propriété de Liste
IMPORT_ACTIONUne action d'import référence un Module source ou une Liste qui a changéMettre à jour le mapping d'import dans le modèle affecté
MODEL_DATA_SOURCEUne connexion de source de données est rompue (référence pipeline ADO). Les sources de données de modèles externes ne sont pas non plus reconnues.Rétablir la connexion dans ADO ou mettre à jour la référence à la source de données

Comment trouver une erreur de formule

Le type d'erreur le plus courant est LINE_ITEM_FORMULA. Voici le workflow de diagnostic :

« Nous vous indiquons généralement le Line Item avec un underscore au début — c'est le code du Line Item en erreur. Allez dans le modèle, recherchez ce code, et cela vous amène directement à l'élément exact. »
— Animateur de l'atelier, jour 1
1

Ouvrir l'Error Log

Depuis l'aperçu de l'Application Framework, cliquez sur le panneau Error Log (en bas à droite). Filtrez sur les erreurs de formule uniquement. Si plus de 10 erreurs, exportez d'abord vers Excel.

2

Noter le code

Chaque erreur inclut un code de Line Item au format _LINEITEMNAMEEXCEPTION (préfixe underscore). Ce code est unique — utilisez-le pour la recherche dans le modèle, pas le nom affiché.

3

Trouver le Line Item affecté

Allez dans le modèle affecté (l'Error Log vous indique lequel). Ouvrez les Line Items. Recherchez le code avec le préfixe underscore. Le résultat de recherche vous mène directement au Module et au Line Item.

4

Comparer au modèle de base

Ouvrez le modèle de base (votre animateur fournira l'accès). Trouvez le même Line Item. Comparez la formule générée à la formule du modèle de base. Identifiez ce qui a changé : renommage de Hiérarchie, suppression de géographie, changement de Dimension.

5

Corriger ou recopier-ajuster

Option A : corrigez la formule directement — mettez à jour les références pour qu'elles correspondent à votre configuration. Option B : copiez la formule du modèle de base vers votre modèle et ajustez les références de Dimensions. L'option B est plus sûre et plus rapide pour la plupart des erreurs LINE_ITEM_FORMULA.

6

Valider

Après la correction, vérifiez que la formule se résout (pas d'indicateur rouge d'erreur de formule). Retournez à l'Error Log et marquez l'erreur comme résolue. Passez à l'erreur suivante.

7

Escalader en cas de blocage

Si l'erreur semble être un problème au niveau plateforme (non causé par votre configuration), notez le code d'erreur et contactez le support Anaplan. Ne passez pas de temps sur les erreurs plateforme pendant une mission de livraison.

La règle des 90 %

ℹ Attendez-vous à des erreurs, livrez systématiquement

Chaque configuration client produira un ensemble d'erreurs différent. L'objectif n'est pas zéro erreur avant de continuer — c'est de passer de 0 % à 90 % rapidement. Catégorisez les erreurs, corrigez systématiquement celles qui le peuvent et consignez les autres comme liste de problèmes connus. « Ces derniers 10 % sont ceux qu'on traite méthodiquement » — ce n'est pas le signe que le projet est cassé.

Exercice sur l'Error Log

1
Exercice sur l'Error Log
⏱ 30 min
Objectif Trouver toutes les erreurs dans le modèle généré. Catégoriser chacune par type d'erreur. Identifier la cause racine d'au moins une erreur LINE_ITEM_FORMULA et proposer une correction.
  1. 1Depuis l'Application Framework, ouvrez l'Error Log. Exportez vers Excel si vous avez plus de 10 erreurs.
  2. 2Créez un tableau simple : Code d'erreur | Type d'erreur | Modèle | Cause racine | Action
  3. 3Pour chaque erreur, catégorisez-la en : LINE_ITEM_FORMULA, LIST_PROPERTY_FORMULA, IMPORT_ACTION ou MODEL_DATA_SOURCE.
  4. 4Choisissez la première erreur LINE_ITEM_FORMULA. Notez le code (préfixe underscore).
  5. 5Allez dans le modèle affecté. Recherchez le code dans les Line Items. Confirmez que vous trouvez le bon Line Item.
  6. 6Ouvrez le Line Item en blueprint. Enregistrez la formule actuelle (cassée).
  7. 7Ouvrez le modèle de base. Trouvez le même Line Item. Enregistrez la formule du modèle de base.
  8. 8Comparez les deux formules. Identifiez quelle référence a changé (renommage de Hiérarchie ? géographie ? Dimension ?).
  9. 9Corrigez la formule dans votre modèle OU recopiez-ajustez depuis le modèle de base. Confirmez qu'elle se résout.
  10. 10Retournez à l'Error Log. Combien d'erreurs restent ? Quelles erreurs peut-on laisser pour l'instant en toute sécurité ?
✅ À quoi cela doit ressembler une fois terminé

Questions de débrief

  1. Quelles erreurs peut-on laisser pour l'instant ? Lesquelles doivent être corrigées avant le test ?
  2. Si un client demande « le modèle est-il prêt à être utilisé ? » et que vous avez 8 erreurs dans le log — que dites-vous ?
  3. Quand escalader une erreur au support Anaplan vs. la corriger soi-même ?
  4. Comment documenteriez-vous votre approche de résolution d'erreurs pour le prochain consultant sur ce projet ?
