🔍 Vérification post-génération — atelier vs. réalité

Cette page couvre les vérifications essentielles à haute priorité pour un modèle fonctionnel.

Dans une implémentation de production réelle, la validation post-génération est bien plus étendue — typiquement 1 à 2 jours de revue systématique couvrant :

  • Vérification de la configuration (structure de Hiérarchies, paramètres de Dimensions)
  • Validation de la sécurité et des accès
  • Optimisation des performances et efficacité des formules
  • Documentation et préparation du transfert
  • Validation du chargement des données (intégration actuals)
  • Scénarios de tests d'acceptation utilisateurs

La checklist post-génération complète se trouve dans la Section 5 et l'Annexe A du Configuration Guide.

Cet atelier se concentre sur les concepts et le workflow de base. Pour des implémentations en production, suivez toujours la checklist complète du Configuration Guide.

Requis après chaque Génération

Après la fin de la Génération, parcourez la checklist suivante avant tout autre travail. Cela garantit que vous disposez d'une base complète et stable avant d'ajouter des données ou des extensions.

  • Vérifier que les 4 modèles existent dans le Workspace (Financial Planning, Headcount, Admin, CapEx)
  • Vérifier le nombre d'objets applicatifs — l'aperçu de l'App Framework affiche : modèles, UX apps, ressources, objets data orchestrator
  • Ouvrir les Error Logs et noter le nombre total d'erreurs — enregistrez-le comme votre baseline
  • Examiner l'Error Log pour vérifier que tous les Modules ont été générés correctement — tout Module manquant apparaîtra ici comme une erreur. Voir la section Revue de l'Error Log pour les détails.
  • Vérifier que les Line Items des Modules SYS clés sont peuplés : la Liste Planning Methods sera pré-remplie. Toutes les Listes de données de production (entity, department, accounts) seront non peuplées et seront alimentées en poussant les données depuis le Data Orchestrator.
  • Confirmer que les Listes de Hiérarchies sont chargées avec les membres attendus. Tous les membres des Listes de Hiérarchies sont peuplés via ADO — certains via CSV ou source externe connectée. Les Hiérarchies comme entity, department, IS Accounts et BS Accounts nécessitent une étape de mapping supplémentaire dans le modèle Admin pour mapper les membres de Listes sources à leurs équivalents de planning avant le push vers d'autres modèles spoke.
  • Vérifier que les membres de la Liste Planning Method sont peuplés — c'est le seul élément vérifiable à ce stade. Les modules SYS et ADMIN liés aux Planning Methods sont peuplés ensuite ; reportez-vous à l'Annexe A3 du Configuration Guide pour compléter cette étape.
  • Effectuer une vérification ponctuelle sur un Line Item calculé dans un Module CALC — confirmer que la formule n'est pas vide
⚠ Formules vides ? Vérifiez d'abord l'Error Log

La meilleure façon de vérifier si une formule devrait être là et s'avère vide est d'examiner l'Error Log. Toutes les erreurs de formule rendent la formule VIDE — elles échouent à cause d'une référence invalide, soit par des changements de noms de Line Items non reconnus, soit par des changements de structure. Vérifiez toujours l'Error Log avant de supposer qu'une formule est manquante.

Mini-exercice post-génération

1
Vérifiez votre modèle généré
⏱ 20 min
Objectif Parcourir la checklist post-génération. Identifier le nombre d'erreurs de base et confirmer que la structure du modèle est complète.
  1. 1Depuis votre Workspace, confirmez que 4 modèles sont listés. Notez leurs noms.
  2. 2Ouvrez la page d'aperçu App Framework. Enregistrez : nombre de modèles, nombre d'UX apps, nombre de ressources, nombre d'erreurs.
  3. 3Cliquez sur Error Logs (en bas à droite). Enregistrez le nombre total d'erreurs — c'est votre baseline.
  4. 4Ouvrez le modèle Financial Planning. Naviguez vers Modules. Vérifiez qu'au moins un Module existe pour chaque préfixe : ADMIN, SYS, DAT, INP, CALC, OUT.
  5. 5Ouvrez un Module SYS. Trouvez la Liste Planning Methods. Vérifiez que les méthodes que vous avez activées dans l'Application Framework Configurator sont présentes.
  6. 6Ouvrez un Module CALC (par ex. CALC OPX IS Itemized ou similaire). Ouvrez un Line Item en blueprint. Vérifiez que le champ formule n'est pas vide.
  7. 7Cochez votre checklist comme complète. Vous êtes prêt à examiner les Error Logs.
