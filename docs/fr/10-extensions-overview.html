Ce que sont les extensions

Les extensions sont des ajouts ou modifications spécifiques à un client au modèle de base IFP que l'Application Framework Configurator ne peut pas produire. Elles entrent dans trois catégories — et la catégorie détermine le risque et l'approche.

Type d'extensionDe quoi il s'agitRisque à la mise à niveauQuand l'utiliser
ConfigurerUtiliser l'Application Framework Configurator pour activer une capacité intégréeAucun — c'est supportéToujours essayer en premier. Si l'Application Framework Configurator a un paramètre pour cela, utilisez l'Application Framework Configurator.
Étendre / ConnecterAjouter de nouveaux Line Items, Modules ou Listes sans toucher aux formules du modèle de baseFaible — les nouveaux éléments ne sont pas écrasés par les mises à niveauQuand le client a besoin de quelque chose que l'Application Framework Configurator ne couvre pas, mais que vous pouvez l'ajouter sans changer la logique du modèle de base.
Amender / Modifier / SupprimerChanger une formule, une Liste ou un Module existant du modèle de baseÉlevé — les changements peuvent être perdus ou cassés à la mise à niveauDernier recours. Documentez chaque override avec un data tag. Maintenez une documentation externe.

7 bonnes pratiques pour les extensions IFP

1

Configurer d'abord

La configuration via l'Application Framework est le moyen principal pour définir la structure d'une application sur la base des entrées et des exigences du client.

2

Étendre ou connecter ensuite

Les extensions devraient toujours être additives, c'est-à-dire ajouter de nouvelles capacités sans modifier la logique existante.

3

Modifier avec précaution

Modifier des objets existants — surtout ceux qui ne peuvent pas être configurés — peut impacter à la fois la configurabilité et la possibilité de mise à niveau.

4

Supprimer avec prudence

Supprimer des composants d'une application post-configurée peut créer un effort supplémentaire lors de futures reconfigurations ou de l'adoption d'une nouvelle version applicative.

5

Maintenir une convention de nommage cohérente

Un nommage cohérent rend les extensions plus faciles à maintenir, supporter et mettre à niveau.

6

Grouper les extensions liées

Regrouper plusieurs extensions les rend plus faciles à identifier et gérer, notamment lors des mises à niveau.

7

Documenter toutes les extensions

Toutes les extensions et modifications doivent être clairement documentées.

Data Tags — créez-les en premier

⚠ Taguez avant de construire

Avant tout changement d'extension sur un modèle IFP, créez deux data tags : Extension (pour tous les nouveaux Line Items que vous ajoutez) et Formula Override (pour toute formule du modèle de base que vous changez). Ces tags sont le seul enregistrement dans le modèle de ce que vous avez modifié. Sans eux, un consultant futur n'a aucun moyen de le savoir.

⚠ Formula Overrides et mises à niveau

Les Formula Overrides sur les Line Items du modèle de base sont l'action d'extension la plus risquée. Lorsque Anaplan publie une mise à jour IFP, les Formula Overrides peuvent être perdus ou cassés. Taguez-les toujours avec le data tag Formula Override et documentez-les en externe. Après toute mise à niveau IFP, filtrez par le tag Formula Override et re-validez chacun.
